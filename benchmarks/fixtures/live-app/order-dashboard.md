# Benchmark Fixture: 주문 대시보드 Live QA

## Fixture ID
`live-order-dashboard`

## 평가 목표

실제로 동작하는 주문 대시보드의 유저 저니를 시뮬레이션하고,
Riff가 숨겨진 UI/UX 버그를 발견하는지 평가합니다.

이 fixture는 코드가 아닌 **유저 저니 시나리오**를 입력으로 사용합니다.
Riff는 이 시나리오를 실행(또는 검토)하며 문제를 찾아야 합니다.

---

## 애플리케이션 설명

**주문 대시보드** — 온라인 쇼핑몰 운영자가 주문을 관리하는 관리자 페이지.

주요 기능:
- 주문 목록 조회 (필터: 상태, 날짜, 고객명)
- 주문 상세 조회 (클릭 시 사이드패널 오픈)
- 배송 처리 (버튼 클릭 → API 호출 → 상태 변경)
- 주문 취소 (사유 입력 필수 → 확인 다이얼로그 → API 호출)

---

## 유저 저니 시나리오

### 시나리오 A: 정상 배송 처리 플로우

1. 운영자가 `/orders` 페이지에 접속
2. 주문 목록에서 상태가 "결제완료"인 주문 클릭
3. 사이드패널이 열리며 주문 상세 표시
4. "배송 처리" 버튼 클릭
5. **[DEFECT-1]** 버튼 클릭 후 아무런 로딩 표시 없이 3~5초 대기
   - 실제로는 API 호출 중이지만 UI 피드백 없음
   - 운영자가 버튼을 여러 번 클릭할 가능성 있음
6. API 응답 후 성공 토스트 표시
7. **[DEFECT-2]** 사이드패널을 닫고 주문 목록을 보면 여전히 "결제완료" 상태
   - 주문 목록이 자동으로 갱신되지 않음 (새로고침해야 반영)
8. 운영자가 같은 주문을 다시 처리할 수도 있음 (중복 처리 위험)

### 시나리오 B: 모바일 환경에서 조회

1. 모바일 브라우저(390px 뷰포트)로 `/orders` 접속
2. **[DEFECT-3]** 주문 목록 테이블이 가로로 잘림
   - 가로 스크롤이 없어서 "총액", "상태" 컬럼이 보이지 않음
   - `overflow-x: auto` 미적용 상태
3. 주문 클릭 시 사이드패널이 전체 화면을 가리지만
   닫기 버튼이 화면 상단 고정이라 스크롤해야 닫을 수 있음

### 시나리오 C: 주문 취소 플로우

1. 상태가 "처리중"인 주문의 "취소" 버튼 클릭
2. 취소 사유 입력 모달 표시
3. 취소 사유 입력 후 "확인" 클릭
4. **[DEFECT-4]** 취소 실패 시 에러 메시지가 단순히 "오류가 발생했습니다"
   - 실제 이유(예: "이미 배송 시작된 주문은 취소 불가")가 표시되지 않음
   - 운영자가 무엇을 해야 하는지 안내 없음
5. 성공 시 주문 상태는 "취소됨"으로 변경되지만
   주문 목록 뱃지 색상이 취소 상태임에도 여전히 초록색 (CSS 클래스 누락)

### 시나리오 D: 대량 주문 처리

1. 100개 이상의 주문이 있는 날 접속
2. 페이지네이션 없이 전체 로드 (스크롤 무한)
3. **[DEFECT-5]** 스크린 리더 사용자의 접근성 문제
   - 주문 테이블에 `aria-label` 없음
   - 상태 뱃지가 색상만으로 구분 (색맹 사용자 고려 없음)
   - "배송 처리" 버튼에 `aria-describedby`로 어느 주문인지 연결 없음
