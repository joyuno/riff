# 5-Layer 질문 트리 — 멀티 프레임워크

각 Layer는 검증된 질문 프레임워크를 적용하며, 답변이 다음 질문과 아키텍처 결정을 결정합니다.

---

## Layer 1: WHY — 이 제품이 해결할 "일" (JTBD)

> 목적: "기능"이 아니라 "Job"에서 시작한다. Job이 명확하면 불필요한 기능이 자연스럽게 탈락한다.
> 프레임워크: Jobs-to-Be-Done — 사용자가 제품을 "고용"해서 시킬 일을 정의

---

### Q1: "이 제품이 없으면, 그 일을 지금 어떻게 하고 있나요? 그 방식의 가장 큰 문제는?"

**왜 이 질문인가:** JTBD의 핵심은 "현재 대안(current solution)"을 파악하는 것입니다. 완전히 새로운 일은 거의 없습니다 — 사용자는 이미 엑셀, 수작업, 경쟁 제품 등으로 그 일을 하고 있습니다. 현재 방식의 문제가 곧 이 제품의 존재 이유입니다.

**답변 분기 및 자동 태깅:**

| 답변 패턴 | 추출 태그 | 아키텍처 영향 |
|---|---|---|
| "엑셀/수작업으로 하는데 너무 느려요" | `performance_critical` | 자동화, 배치 처리, 비동기 처리 설계 |
| "사람이 하다 보니 실수가 계속 나요" | `reliability_critical` | 검증 레이어, 감사 로그, 롤백 메커니즘 |
| "아직 아무도 안 하고 있어요" | `market_risk` | 빠른 출시, 피벗 가능한 구조, 유연한 스키마 |
| "경쟁사 제품이 있지만 불편해요" | `competitive_pressure` | 차별화 기능 집중, UX 우선 |
| "규정 때문에 꼭 필요해요" | `compliance_required` | 감사 추적, 데이터 보존, 접근 제어 |

**답변이 모호할 때 (JTBD 심화):**
- "그 일을 하는 데 일주일에 몇 시간을 쓰고 있나요?"
- "그 방식이 가장 짜증나는 순간을 하나만 말해주세요."
- "만약 마법처럼 그 문제가 사라진다면, 그 시간에 뭘 하고 싶으세요?"

---

### Q2: "이 제품을 '고용'한 사용자가 하루가 끝날 때 '오늘 이걸 써서 다행이다'라고 느끼는 순간은 어떤 순간인가요?"

**왜 이 질문인가:** JTBD에서 "desired outcome"을 정의합니다. 기능 목록이 아닌 **감정적 결과**를 먼저 정의하면, MVP에 진짜 필요한 것만 남습니다. "주문을 놓치지 않아서 안심했다"와 "매출 리포트가 예쁘게 나왔다"는 완전히 다른 제품입니다.

**답변 처리:**

| 감정적 결과 | Job 유형 | MVP 방향 |
|---|---|---|
| "안심", "걱정 없이" | 안정성/신뢰성 Job | 알림, 모니터링, 자동 백업 중심 |
| "빨라졌다", "시간 절약" | 효율성 Job | 자동화, 대시보드, 배치 처리 중심 |
| "돈을 벌었다", "매출이 올랐다" | 수익 Job | 전환 최적화, 분석, 결제 중심 |
| "쉬워졌다", "누구나 할 수 있게" | 접근성 Job | UX 단순화, 온보딩, 가이드 중심 |
| "새로운 걸 할 수 있게 됐다" | 혁신 Job | 핵심 차별화 기능, 빠른 실험 중심 |

**자동 메모:** Q2의 감정적 결과를 기록하고, Layer 5 Pre-mortem에서 "이 감정을 못 느끼게 되는 시나리오"로 연결합니다.

---

## Layer 2: WHO — 실제 행동 기반 사용자 (Mom Test)

> 목적: 가설이 아닌 실제 행동에서 사용자를 파악한다. "쓸 것 같아요"는 거짓말이다.
> 프레임워크: Mom Test — 엄마한테 물어도 진짜 답이 나오는 질문만 한다

---

### Q3: "이 문제를 겪는 사람을 한 명 떠올려보세요. 그 사람이 지난주에 이 문제를 어떻게 해결했나요?"

**왜 이 질문인가:** Mom Test의 핵심 원칙 — "과거에 실제로 한 행동"을 묻는 것입니다. "어떤 기능을 원하세요?"는 상상의 답을 유도하지만, "지난주에 어떻게 했나요?"는 사실만 나옵니다.

**답변 분기:**

| 사용자 프로필 패턴 | 추출 정보 | 설계 영향 |
|---|---|---|
| 구체적 인물 묘사 + 행동 | 명확한 페르소나 | 해당 페르소나 기준으로 UX 설계 |
| "저요, 제가 직접 쓸 거예요" | 자기 자신이 사용자 | 가장 정확한 피드백 루프 가능 |
| "글쎄요, 잘 모르겠어요" | 사용자 미정 | ⚠️ 경고: 사용자 없는 제품은 위험. 추가 질문 필요 |

**Mom Test 리디렉션 규칙:**
- ❌ "~할 것 같아요", "~하면 좋을 것 같아요" → ✅ "실제로 그렇게 해본 적이 있나요?"
- ❌ "다들 필요할 거예요" → ✅ "구체적으로 누가 필요하다고 했나요? 그 대화를 기억나세요?"
- ❌ "좋은 아이디어인 것 같아요" → ✅ "이 문제 때문에 돈을 쓰거나 시간을 낭비한 적이 있나요?"

---

### Q4: "그 과정에서 가장 짜증났던 순간은 언제였나요? 구체적인 상황으로요. 돈이나 시간을 얼마나 낭비했나요?"

**왜 이 질문인가:** Mom Test에서 "problem severity"를 측정합니다. 짜증의 강도와 빈도가 곧 제품의 가치입니다. 돈/시간 낭비를 물으면 "있으면 좋겠다" 수준인지 "없으면 죽는다" 수준인지가 드러납니다.

**예시 답변과 영향:**

```
답변: "매주 금요일마다 3시간씩 엑셀 복붙해요. 한 번 실수하면 주말에 다시 해야 해요."
→ 주 3시간 × 52주 = 연 156시간 낭비. 높은 가치. 자동화가 핵심 Job.

답변: "가끔 불편하긴 한데... 뭐 참을 만해요."
→ ⚠️ 낮은 고통. 이 문제를 해결해도 사용자가 전환하지 않을 수 있음.
   리디렉션: "그 불편함 때문에 다른 방법을 찾아본 적은 있나요?"
```

---

### Q5: "이 서비스를 동시에 쓸 사람이 몇 명 정도 될까요?"

**왜 이 질문인가:** 아키텍처의 기반이 되는 질문입니다. 10명용 시스템과 10,000명용 시스템은 설계가 완전히 다릅니다.

**선택지 및 아키텍처 영향:**

| 선택지 | 규모 | 추천 아키텍처 | 인프라 |
|---|---|---|---|
| A: 1~10명 | 팀 내부 도구 | 모놀리식, 단일 서버 | 단일 서버 또는 Railway, SQLite 가능 |
| B: 100~1,000명 | 중소 서비스 | 모놀리식 + CDN | RDS + 캐싱, Vercel/Render |
| C: 10,000명 이상 | 공개 서비스 | 수평 확장 가능 구조 | 로드밸런서, DB 분리, 캐싱 필수 |

**후속 질문 (C 선택 시):**
- "그 사용자 중 동시에 글을 쓰거나 주문을 하는 비율은 얼마나 될까요?" (읽기 vs 쓰기 비율 → DB 선택 영향)

## Layer 3: WHAT — 제약 조건으로 핵심 찾기 (Constraint Forcing)

> 목적: 기능 욕심을 제거하고 진짜 핵심을 찾는다. "다 중요해요"는 아무것도 중요하지 않다는 뜻이다.
> 프레임워크: Constraint Forcing + 5 Whys — 인위적 제약으로 발산을 수렴시킨다

---

### Q6: "극단적으로 단순화해볼게요. 이 서비스에서 딱 하나의 화면, 하나의 버튼만 있다면 그게 뭔가요?"

**왜 이 질문인가:** "모든 기능을 나열해주세요"는 발산만 일으킵니다. 극단적인 제약("하나만")을 걸면 진짜 핵심이 자동으로 드러납니다. 이 답변이 곧 MVP의 핵심 가치입니다.

**점진적 확장 프로세스:**

```
Step 1: "딱 하나만" → 핵심 Job 확인
  예: "상품 주문 확인 버튼" → 핵심 Job = 주문 관리

Step 2: "거기에 2~3개 더 추가한다면?" → 범위 확장
  → 사용자가 직접 우선순위를 정하게 됨

Step 3: 분류 확정
  - 반드시 있어야 함: [최대 3개]
  - 있으면 좋음: [최대 3개]
  - 나중에: [나머지]
```

**일관성 검사:**
- Layer 1 태그가 `performance_critical`인데 "반드시"에 속도 관련 기능이 없으면 → 모순 지적
- Layer 2 Q5가 C(10,000명)인데 "반드시"에 실시간 채팅이 있으면 → 복잡도 경고
- "반드시" 목록이 4개 이상이면 → "하나만 빼야 한다면 어떤 걸 빼시겠어요?"로 재수렴

---

### Q7: "첫 버전을 언제까지 써보고 싶으세요?"

**왜 이 질문인가:** 시간 제약이 기술 선택과 품질 수준을 결정합니다. 2주 안에 나와야 하면 새로운 기술 학습은 불가능합니다.

**선택지 및 Pulse 전략:**

| 타임라인 | Pulse 전략 | 에이전트 구성 |
|---|---|---|
| A: 2주 이내 | 속도 최우선: 검증된 스택, 최소 기능 | 에이전트 최대화, 병렬화, QA Tier 1 |
| B: 1~2개월 | 균형: 표준 팀 구성 | 표준 에이전트, 핵심 테스트, QA Tier 2 |
| C: 3개월 이상 | 품질 우선: 테스트 강화, 문서화 | 전문화 에이전트, 풀 QA, QA Tier 3 |

**시간-범위 충돌 감지:**
- Q7=A(2주) + Q6 반드시 3개 + 복잡 기능 → "2주 안에 이 세 개를 다 하기 어렵습니다. 우선순위를 한 번 더 정해볼까요?"

## Layer 4: HOW — 트레이드오프 탐색 (소크라테스)

> 목적: 기술 결정의 비용을 이해하고, 사용자가 직접 선택하게 한다.
> 프레임워크: 소크라테스 대화법 — "가르치지 않고 질문으로 깨닫게 한다"

---

### Q8: "기술에 대해 잘 모르셔도 괜찮습니다. 이미 사용하고 있거나 익숙한 도구가 있나요?"

**왜 이 질문인가:** 비개발자에게 "React vs Vue"를 묻는 건 무의미합니다. 대신 익숙한 도구에서 출발합니다. "노션을 많이 써요" → 노션 같은 UI, "엑셀로 다 해요" → 테이블 중심 인터페이스.

**답변 처리:**

```
기술 경험 있음 → 해당 스택 기준으로 Q9 생성
  예: "Python 좀 할 줄 알아요" → Django/FastAPI 선택지 제시

비개발자 → 사용 패턴에서 기술 추론
  예: "노션으로 관리해요" → 구조화된 데이터, 테이블 중심 → 관리형 서비스 추천

완전 미정 → 도메인 + Q5 답변 기반으로 스택 추천
  예: 웹 + B규모 → "Next.js + Supabase를 추천드려요. 이유는..."
```

---

### Q9: [Layer 1 태깅 기반 동적 생성 — 소크라테스 대화]

**왜 이 질문인가:** 일반적인 기술 질문이 아닌, 이 프로젝트의 핵심 트레이드오프를 비개발자 언어로 탐색합니다.

**태그별 소크라테스 질문:**

**`performance_critical` 태그:**
> "속도가 중요하다고 하셨죠. 두 가지 방향이 있어요:
> A) 한 사람이 버튼을 누르면 0.5초 안에 결과가 나오는 것
> B) 100명이 동시에 눌러도 안 터지는 것
> 어떤 상황이 더 가까운가요?
> _(A면 캐싱/최적화, B면 수평 확장 설계)_"

**`reliability_critical` 태그:**
> "실수가 문제라고 하셨죠. 두 가지 접근이 있어요:
> A) 실수를 아예 못하게 막는 것 (입력할 때 검증, 확인 버튼 2번)
> B) 실수하더라도 '되돌리기'로 바로 복구하는 것
> 어느 쪽이 이 상황에 더 맞나요?
> _(A면 검증 레이어, B면 버전 관리/감사 로그)_"

**`market_risk` 태그:**
> "새로운 시장이니 빠르게 테스트하는 게 중요하겠네요. 두 가지 길이 있어요:
> A) 2주 안에 나오지만, 사용자가 많아지면 구조를 바꿔야 할 수 있음
> B) 한 달 걸리지만, 나중에 확장할 때 처음부터 다시 안 만들어도 됨
> 지금 단계에서 어느 쪽이 더 중요한가요?"

**`compliance_required` 태그:**
> "규정 준수가 필요하시군요. 어떤 종류의 규정인지에 따라 설계가 달라져요:
> A) 개인정보보호 (누가 무엇을 봤는지 기록 필요)
> B) 금융 규정 (거래 기록 보존, 감사 추적)
> C) 의료/건강 데이터 (암호화, 접근 제한)
> 해당되는 게 있나요?"

**`competitive_pressure` 태그:**
> "경쟁사가 있다고 하셨죠. 경쟁에서 이기려면 '모든 것을 조금씩 더 잘하는 것'보다 '한 가지를 압도적으로 잘하는 것'이 효과적이에요.
> 속도 / 가격 / 사용 편의성 / 특정 기능 중에서, 딱 하나만 고르면 어디서 이기고 싶으세요?"

---

## Layer 5: MEASURE — 실패 시나리오 사전 제거 (Pre-mortem)

> 목적: "성공 기준을 정해주세요"는 추상적인 답만 나온다. 실패를 먼저 상상하면 구체적인 기준이 나온다.
> 프레임워크: Pre-mortem — "실패했다고 치자. 왜 실패했나?"

---

### Q10: "한번 상상해볼게요. 3개월 후, 이 서비스를 만들었는데 실패했어요. 가장 가능성 높은 이유가 뭘까요?"

**왜 이 질문인가:** Pre-mortem은 "성공하려면 뭘 해야 할까?"보다 **"실패한다면 왜일까?"**가 더 구체적인 답을 끌어낸다는 연구 결과에 기반합니다. 사람들은 성공을 상상할 때는 막연하지만, 실패를 상상할 때는 구체적입니다.

**답변 분기 및 자동 처리:**

| 실패 시나리오 | 추출 리스크 | 자동 대응 |
|---|---|---|
| "아무도 안 쓸 것 같아요" | `adoption_risk` | MVP에 온보딩/첫 경험 최적화 포함 |
| "기술적으로 안 될 것 같아요" | `technical_risk` | 핵심 기술 PoC를 첫 Pulse에 배치 |
| "경쟁사가 더 빠를 것 같아요" | `speed_risk` | 타임라인 A 강제, 차별화 기능 집중 |
| "돈이 너무 많이 들 것 같아요" | `cost_risk` | 무료/저가 인프라 우선, 관리형 서비스 |
| "만들어도 내가 원하던 게 아닐 것 같아요" | `scope_risk` | Pulse 주기를 짧게, 매 사이클 사용자 확인 |

**Immunity 연동:** Q10의 실패 시나리오는 Pulse Immunity 시스템의 **초기 항체**로 자동 등록됩니다.

---

### Q10-후속: "그 실패를 막으려면, 출시 첫 달에 어떤 숫자를 봐야 '잘 되고 있다'고 확신할 수 있을까요?"

**왜 이 질문인가:** 실패 시나리오를 뒤집으면 성공 기준이 됩니다. "아무도 안 쓸까봐" → "첫 달에 DAU 100명이면 안심"처럼, 실패 공포에서 구체적인 숫자가 나옵니다.

**선택지:**
- A: 사용량 지표 (DAU, 요청 수, 처리 건수)
- B: 비즈니스 지표 (매출, 전환율, 고객 유지율)
- C: 품질 지표 (오류율, 응답시간, 정확도)

**수치 없는 답변 처리:**
> "조금 더 구체적으로 해볼까요? [Q10 실패 시나리오]가 일어나지 않았다는 걸 어떤 숫자로 확인할 수 있을까요? 예를 들면 [예시 수치]처럼요."

---

### Q10-리스크: "방금 말씀하신 실패 시나리오를 막기 위해, MVP에서 반드시 검증해야 할 가설은 뭘까요?"

**왜 이 질문인가:** Pre-mortem의 마무리 — 실패 시나리오에서 **검증 가능한 가설**을 추출합니다. 이 가설이 Pulse QA의 테스트 시나리오가 됩니다.

**예시:**
```
실패 시나리오: "아무도 안 쓸 것 같아요"
→ 검증 가설: "타겟 사용자 10명에게 보여주면 3명 이상이 '이거 쓰고 싶다'고 할 것이다"
→ QA 시나리오: 첫 사용자 경험 테스트 (온보딩 → 핵심 기능 → 만족도)

실패 시나리오: "기술적으로 안 될 것 같아요"
→ 검증 가설: "핵심 API가 1초 이내에 응답할 수 있다"
→ QA 시나리오: 성능 테스트 (Tier 3 — 부하 테스트)
```
