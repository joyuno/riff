### SS-Q1: 판매 채널 및 우선순위

**질문:**
> "어떤 판매 채널에서 시작하실 건가요? 쿠팡, 네이버스마트스토어, 11번가, 아마존, 자체몰 중 어느 채널부터 자동화하고 싶으신가요? 그리고 동시에 여러 채널을 관리해야 하나요?"

**왜 이 질문이 중요한가:**
채널마다 API가 다르고, 정책이 다르고, 수수료 구조가 다릅니다. 멀티채널은 단일 채널 자동화보다 3-5배 복잡합니다. 첫 MVP는 한 채널로 시작하는 게 맞습니다.

**관련 Layer:** WHAT, HOW

**답변 분기:**

| 채널 조합 | 자동화 복잡도 | 핵심 API | MVP 전략 |
|---|---|---|---|
| 네이버 스마트스토어만 | 낮음 | 네이버 커머스 API | 상품 등록 + 주문 처리 먼저 |
| 쿠팡만 | 낮음 | 쿠팡 Wing API | 상품 등록 + 가격 모니터링 먼저 |
| 네이버 + 쿠팡 | 중간 | 두 API 연동 | 상품 정보 표준화 레이어 필요 |
| 3개 이상 채널 | 높음 | 멀티 API + 정규화 | 채널 추상화 레이어 설계 필수 |
| 아마존 포함 | 높음 | MWS / SP-API | 국제 배송, 환율, 세금 처리 추가 |

### SS-Q2: 상품 소싱 방식

**질문:**
> "상품을 어떻게 소싱하시나요? 직접 사입해서 재고를 보유하나요, 위탁(드롭쉬핑)으로 재고 없이 판매하나요, 아니면 국내 도매 사이트에서 자동으로 긁어오나요?"

**왜 이 질문이 중요한가:**
소싱 방식이 재고 관리 로직의 복잡도를 결정합니다. 사입은 재고 수량 추적이 핵심이고, 위탁은 공급사 재고 실시간 반영이 핵심입니다.

**관련 Layer:** WHY, HOW

**답변 분기:**

| 소싱 방식 | 자동화 핵심 | 기술 요구사항 |
|---|---|---|
| 사입 (재고 보유) | 재고 수량 실시간 추적, 발주 자동화 | 재고 DB + 입출고 로직 |
| 위탁 (드롭쉬핑) | 공급사 재고 폴링, 품절 자동 처리 | 공급사 사이트 크롤링 또는 API |
| 도매 사이트 연동 | 상품 정보 수집 + 마진 계산 | 크롤러 + 이미지 처리 + 가격 계산 |
| 해외 소싱 (중국 등) | 환율 반영 가격 계산, 배송비 자동화 | 환율 API + 배송비 매트릭스 |

### SS-Q3: 자동화 범위 정의

**질문:**
> "자동화하고 싶은 작업을 우선순위대로 골라주세요. 상품 등록 / 가격 자동 조정 / 재고 관리 / 주문 처리 및 배송 / CS(문의) 자동화 / 정산 관리 중에서 지금 가장 많은 시간을 잡아먹는 작업이 무엇인가요?"

**왜 이 질문이 중요한가:**
모든 걸 한 번에 자동화하려다가 아무것도 제대로 못 하는 경우가 많습니다. 가장 고통받는 작업 하나를 완벽히 자동화하는 것이 MVP의 목표입니다.

**관련 Layer:** WHAT

**답변 분기:**

| 우선 자동화 대상 | 예상 시간 절감 | 구현 복잡도 |
|---|---|---|
| 상품 등록 | 건당 30분 → 2분 | 중간 (상품 정보 표준화, 이미지 처리) |
| 가격 자동 조정 | 매일 수동 확인 제거 | 낮음 (경쟁사 모니터링 + 규칙 엔진) |
| 재고 관리 | 품절 사고 방지 | 중간 (멀티 채널 재고 동기화) |
| 주문 처리/배송 | 주문당 10분 → 1분 | 높음 (택배사 API 연동) |
| CS 자동화 | 반복 문의 80% 자동 처리 | 높음 (LLM + 템플릿 매핑) |
| 정산 관리 | 월말 정산 자동화 | 높음 (채널별 정산 포맷 통합) |

### SS-Q4: 마진 구조와 가격 전략

**질문:**
> "가격 결정 방식을 설명해주세요. 원가에 고정 마진율을 붙이는 방식인가요, 경쟁사 가격을 보고 실시간으로 조정하나요, 아니면 특정 조건(재고 수준, 시즌)에 따라 다르게 가나요?"

**왜 이 질문이 중요한가:**
가격 결정 로직의 복잡도가 시스템의 핵심 알고리즘을 결정합니다. 단순 마진율은 쉽지만, 동적 가격은 경쟁사 모니터링 + 의사결정 엔진이 필요합니다.

**관련 Layer:** HOW

**답변 분기:**

| 가격 전략 | 구현 방식 | 추가 필요 |
|---|---|---|
| 고정 마진율 (원가 × 1.3) | 단순 계산 공식 | 원가 DB만 있으면 됨 |
| 경쟁사 가격 추종 | 크롤러 + 가격 규칙 엔진 | 주기적 경쟁사 모니터링 |
| 동적 가격 (재고/시즌 반영) | 가격 최적화 알고리즘 | 규칙 + 데이터 분석 |
| 최저가 보장 + 마진 하한 | 범위 기반 조정 로직 | 최저가/최고가 한도 설정 |

### SS-Q5: 상품 수와 카테고리 범위

**질문:**
> "현재 또는 목표로 하는 상품 수는 얼마나 되나요? 100개 이하인가요, 1,000개, 아니면 10,000개 이상인가요? 그리고 카테고리가 다양한가요, 특정 카테고리에 집중하나요?"

**왜 이 질문이 중요한가:**
상품 수가 시스템 규모와 성능 요구사항을 결정합니다. 10만 개 상품 동기화는 단순 루프로 처리하면 수 시간이 걸립니다.

**관련 Layer:** WHO (규모)

**답변 분기:**

| 상품 수 | 아키텍처 | 처리 방식 |
|---|---|---|
| 100개 이하 | 단순 배치 | 주기적 동기화 (1시간마다) |
| 1,000~10,000개 | 배치 + 우선순위 큐 | 변경 감지 후 우선 처리 |
| 10,000개 이상 | 분산 처리 + 캐싱 | 병렬 처리, Delta 동기화 |

### SS-Q6: 경쟁 분석 자동화 필요 여부

**질문:**
> "경쟁 셀러나 경쟁 상품을 모니터링해야 하나요? 예를 들어 특정 키워드 순위, 경쟁사 가격 변동, 신상품 입고 알림 같은 것들이 필요한가요?"

**왜 이 질문이 중요한가:**
경쟁 분석은 크롤링 기반이라 법적/기술적 리스크가 있습니다. 공식 API가 없는 사이트 크롤링은 IP 차단, 봇 감지 등의 리스크를 고려해야 합니다.

**관련 Layer:** WHAT, HOW

**답변 분기:**

| 필요 수준 | 구현 방법 | 리스크 |
|---|---|---|
| 필요 없음 | — | — |
| 가격만 모니터링 | 공식 API 우선, 크롤러 보조 | 낮음 |
| 키워드 순위 추적 | 네이버 검색 API 또는 크롤러 | 중간 (IP 로테이션 필요) |
| 전체 경쟁 모니터링 | Playwright 기반 크롤러 + 프록시 | 높음 (서비스 약관 확인 필수) |

### SS-Q7: 주요 API 연동 및 택배사 연동

**질문:**
> "어떤 외부 시스템과 연동해야 하나요? 택배사(CJ대한통운, 한진, 롯데), 발주 ERP, 세금계산서 발행, 카카오 알림톡 같은 것들이요. 각각 이미 사용 중인 서비스가 있나요?"

**왜 이 질문이 중요한가:**
연동해야 할 외부 시스템 수가 개발 기간을 직접 결정합니다. API가 없는 시스템은 크롤링이나 수동 처리가 필요합니다.

**관련 Layer:** HOW

**주요 API 현황:**

| 시스템 | API 상태 | 난이도 |
|---|---|---|
| 네이버 커머스 API | 공식 API 있음 | 낮음 |
| 쿠팡 Wing API | 공식 API 있음 | 낮음 |
| CJ대한통운 | 공식 API 있음 | 낮음 |
| 한진택배 | 공식 API 있음 | 낮음 |
| 카카오 알림톡 | 공식 API 있음 | 낮음 |
| 홈택스 세금계산서 | API 없음 (크롤링) | 높음 |
| 도매꾹/도매토피아 | 비공식 크롤링 | 높음 |

### SS-Q8: 마케팅 자동화 필요 여부

**질문:**
> "상품 등록 이후 마케팅 자동화도 필요한가요? 네이버 키워드 광고 자동 입찰, 상품 리뷰 요청 자동화, SNS 자동 포스팅 같은 것들이요."

**왜 이 질문이 중요한가:**
마케팅 자동화는 운영 자동화와 완전히 다른 도메인입니다. 초기 MVP에 포함하면 범위가 너무 커집니다.

**관련 Layer:** WHAT

**답변 분기:**

| 답변 | 권장 |
|---|---|
| 지금은 필요 없음 | MVP에서 제외, 2차에서 고려 |
| 리뷰 요청 자동화만 | 주문 완료 후 알림톡 발송 (간단) |
| 광고 자동 입찰 | 네이버/쿠팡 광고 API 별도 프로젝트 권장 |
| SNS 자동화 | Meta API / 카카오 채널 API, 별도 모듈 |

## 도메인 특화 리스크

| 리스크 | 수준 | 설명 |
|---|---|---|
| 쇼핑몰 API 정책 변경 | 🔴 높음 | 네이버/쿠팡 API는 수시로 바뀜. 버전 관리와 알림 구독 필수 |
| 크롤링 IP 차단 | 🔴 높음 | 공식 API 없는 사이트 크롤링은 IP 차단 위험. 프록시 로테이션 설계 필요 |
| 재고 오류로 인한 품절 사고 | 🔴 높음 | 재고 동기화 오류 시 판매 불가 상품 노출. 멱등성(idempotency) 보장 필수 |
| 가격 계산 버그 | 🟡 중간 | 마진 계산 오류 시 손해 판매 발생. 테스트 케이스 반드시 작성 |
| 정산 오차 | 🟡 중간 | 채널별 정산 주기와 수수료 구조가 다름. 회계 검증 레이어 필요 |
| 세금계산서 자동화 | 🟢 낮음 | 홈택스 API 미제공. 수동 처리 또는 유료 솔루션 사용 권장 |
