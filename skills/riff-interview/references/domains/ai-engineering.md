### AE-Q1: 프로젝트 유형 분류

**질문:**
> "만들려는 AI 시스템이 다음 중 어느 유형에 가장 가까운가요? ① LLM API를 호출해서 기능을 만드는 LLM 앱, ② 특정 데이터로 모델을 추가 학습하는 파인튜닝, ③ 회사 문서를 기반으로 질의응답하는 RAG, ④ 도구를 사용해 작업을 자율 수행하는 에이전트, ⑤ 모델 학습/배포/모니터링 자동화 MLOps"

**왜 이 질문이 중요한가:**
AI 프로젝트 유형에 따라 필요한 인프라, 비용 구조, 개발 기간이 완전히 달라집니다. "AI 챗봇 만들기"라는 같은 요구사항도 RAG와 파인튜닝은 접근법이 정반대입니다.

**관련 Layer:** WHY, HOW

**유형별 특성:**

| 유형 | 개발 기간 | 인프라 비용 | 특화 지식 | 권장 시작점 |
|---|---|---|---|---|
| LLM 앱 | 빠름 (1-4주) | API 비용만 | 낮음 | MVP 첫 선택 |
| RAG | 중간 (2-6주) | API + 벡터DB | 중간 | 사내 지식베이스 |
| 에이전트 | 중간~느림 (4-12주) | API + 실행 환경 | 중간~높음 | 반복 작업 자동화 |
| 파인튜닝 | 느림 (4-16주) | GPU 서버 | 높음 | 특정 도메인 특화 |
| MLOps | 매우 느림 (8주+) | 인프라 전반 | 매우 높음 | 모델 운영 성숙도 높을 때 |

### AE-Q2: 모델 선택 기준

**질문:**
> "어떤 AI 모델을 사용할 건가요? GPT-4o / Claude Sonnet 같은 상용 API, Llama / Mistral 같은 오픈소스 자체 호스팅, 아니면 특정 태스크 특화 소형 모델을 고려하고 있나요? 가장 중요한 선택 기준은 비용인가요, 성능인가요, 데이터 보안인가요?"

**왜 이 질문이 중요한가:**
모델 선택이 비용, 개인정보 처리 방식, 성능을 결정합니다. 특히 기업 내부 문서나 개인정보가 있는 경우 상용 API에 데이터를 보내는 것이 정책상 허용되는지 먼저 확인해야 합니다.

**관련 Layer:** HOW

**모델 선택 매트릭스:**

| 우선 기준 | 추천 선택 | 비용 (추정) | 주의사항 |
|---|---|---|---|
| 최고 성능 | GPT-4o / Claude Opus | $15~60/1M 토큰 | API 비용 모니터링 필수 |
| 비용 최적화 | GPT-4o-mini / Claude Haiku | $0.15~0.6/1M 토큰 | 복잡한 추론 한계 |
| 데이터 보안 | Llama 3 자체 호스팅 | GPU 서버 비용 | 운영 부담 |
| 한국어 특화 | HyperCLOVA X / EXAONE | 네이버/LG 계약 필요 | 한국어 품질 우수 |
| 빠른 응답 | Groq (Llama 3 호스팅) | $0.05~0.09/1M 토큰 | 컨텍스트 제한 |

**데이터 보안 질문 (기업 대상):**
> "처리하는 데이터에 개인정보, 영업비밀, 의료정보, 금융정보가 포함되나요? 포함된다면 상용 API를 쓰기 전에 개인정보처리방침과 API 제공사의 데이터 처리 약관을 반드시 확인해야 합니다."

### AE-Q3: RAG 설계 (해당 시)

**질문 (유형이 RAG일 때):**
> "검색 대상 문서의 특성을 말씀해주세요. 문서 수는 얼마나 되나요? PDF/Word 같은 정형 문서인가요, 슬랙/이메일 같은 비정형 텍스트인가요? 그리고 최신 문서가 자동으로 인덱싱되어야 하나요, 아니면 수동으로 업로드하나요?"

**왜 이 질문이 중요한가:**
문서 특성이 청킹(Chunking) 전략, 임베딩 모델, 벡터DB 선택을 결정합니다. PDF 파싱은 생각보다 복잡하고, 실시간 인덱싱은 파이프라인 설계가 필요합니다.

**관련 Layer:** HOW

**RAG 설계 결정 트리:**

| 문서 특성 | 청킹 전략 | 임베딩 모델 | 벡터DB 추천 |
|---|---|---|---|
| 100개 이하 PDF | 고정 청크 (500토큰) | OpenAI ada-002 | pgvector (PostgreSQL) |
| 1,000개 이상 문서 | Semantic Chunking | OpenAI / Cohere | Pinecone 또는 Weaviate |
| 코드 문서 | 함수/클래스 단위 | code-embedding 특화 | Chroma 또는 pgvector |
| 실시간 업데이트 | 이벤트 기반 파이프라인 | 일관된 모델 유지 | Weaviate (자동 업데이트) |
| 한국어 문서 | 형태소 분석 고려 | multilingual-e5 | pgvector |

**벡터DB 비용 비교:**

| DB | 호스팅 | 1M 벡터 비용 | 추천 상황 |
|---|---|---|---|
| pgvector | 자체 (Supabase) | 무료~$25/월 | 소규모, PostgreSQL 이미 사용 시 |
| Chroma | 자체 호스팅 | 서버 비용만 | 로컬 개발, 프로토타입 |
| Pinecone | 클라우드 | $70~/월 | 대규모, 관리 부담 없애고 싶을 때 |
| Weaviate | 클라우드/자체 | $25~$95/월 | 멀티모달, 자동 업데이트 필요 시 |

### AE-Q4: 에이전트 아키텍처 (해당 시)

**질문 (유형이 에이전트일 때):**
> "에이전트가 어떤 도구(Tool)를 사용해야 하나요? 웹 검색, 코드 실행, 파일 읽기/쓰기, 외부 API 호출, DB 쿼리 중 어느 것인가요? 그리고 에이전트가 자율적으로 결정해야 하는 범위는 어디까지인가요? 인간 승인이 필요한 단계가 있나요?"

**왜 이 질문이 중요한가:**
에이전트의 도구 범위와 자율성 수준이 안전장치 설계를 결정합니다. "모든 걸 자율적으로"는 현재 LLM 수준에서 위험합니다. 중요한 결정에는 반드시 Human-in-the-loop를 설계해야 합니다.

**관련 Layer:** HOW

**에이전트 패턴:**

| 패턴 | 설명 | 적합한 상황 | 위험도 |
|---|---|---|---|
| ReAct (추론+행동) | 생각→행동→관찰 루프 | 범용 작업 | 중간 |
| Plan-and-Execute | 계획 먼저, 실행 나중 | 복잡한 다단계 작업 | 낮음 |
| Multi-Agent | 전문 에이전트 협력 | 대규모 복잡 작업 | 높음 |
| Human-in-the-loop | 중요 단계 인간 승인 | 돌이킬 수 없는 작업 | 매우 낮음 |

**Human-in-the-loop 필수 케이스:**
- 이메일/메시지 발송
- 파일 삭제/수정
- 금전 관련 작업
- 외부 API 호출 (특히 쓰기 작업)

### AE-Q5: 프롬프트 엔지니어링 vs 파인튜닝 결정

**질문:**
> "지금 상용 LLM으로 테스트해봤을 때 결과가 어떤가요? 기본 모델로는 원하는 출력 형식이나 품질이 나오지 않는다면, 그 이유가 ① 프롬프트가 부족해서인가요, ② 특정 도메인 지식이 없어서인가요, ③ 출력 형식/스타일이 맞지 않아서인가요?"

**왜 이 질문이 중요한가:**
파인튜닝은 비용(GPU 시간 + 데이터 준비)이 크고 기간이 깁니다. 프롬프트 엔지니어링으로 해결 가능한 문제를 파인튜닝으로 접근하는 것은 낭비입니다.

**관련 Layer:** HOW

**결정 가이드:**

| 상황 | 권장 접근 | 이유 |
|---|---|---|
| 기본 LLM으로 70%는 됨 | 프롬프트 최적화 먼저 | 파인튜닝 전 최대한 프롬프트로 해결 |
| 특정 형식 출력 필요 | Few-shot 프롬프팅 | 예시 제공만으로 대부분 해결 |
| 도메인 전문 지식 부족 | RAG 먼저 | 파인튜닝 없이도 지식 주입 가능 |
| 특정 스타일/페르소나 필요 | 파인튜닝 고려 | 수백~수천 개 예시 데이터 필요 |
| 추론 성능 자체 개선 | 더 큰 모델 사용 | 파인튜닝으로 추론 능력은 안 올라감 |

### AE-Q6: 평가 메트릭 및 품질 관리

**질문:**
> "AI 출력 품질을 어떻게 평가할 건가요? 자동화된 메트릭(BLEU, ROUGE, 정확도)을 쓸 건가요, 사람이 직접 평가하는 Human Eval을 쓸 건가요, 아니면 LLM이 LLM을 평가하는 LLM-as-a-Judge 방식을 쓸 건가요? 그리고 응답 지연시간과 비용 한도는 얼마인가요?"

**왜 이 질문이 중요한가:**
AI 시스템에서 "좋은 출력"의 기준이 없으면 개선이 불가능합니다. 특히 생산 배포 후 품질 저하를 자동으로 감지하는 시스템이 없으면 문제를 뒤늦게 발견합니다.

**관련 Layer:** MEASURE

**평가 방식 비교:**

| 방식 | 비용 | 확장성 | 정확도 | 적합한 상황 |
|---|---|---|---|---|
| 자동화 메트릭 | 낮음 | 높음 | 낮음 (번역 등 일부만) | CI/CD 파이프라인 |
| Human Eval | 높음 | 낮음 | 높음 | 릴리즈 전 최종 검증 |
| LLM-as-a-Judge | 중간 | 높음 | 중간 | 자동화 + 의미 평가 |
| A/B 테스트 | 중간 | 중간 | 높음 (실사용 기반) | 프로덕션 실험 |

**비용/지연 기준 설정:**
- 응답 지연: 일반 챗봇은 3초 이내, 실시간 스트리밍이면 첫 토큰 1초 이내
- 비용 한도: 사용자당 하루 최대 API 비용을 설정해야 비용 폭증 방지 가능

### AE-Q7: 배포 형태 및 스케일링

**질문:**
> "AI 기능을 어떻게 서비스할 건가요? 사용자가 직접 접근하는 API 서비스인가요, 기존 시스템에 임베딩되는 내부 모듈인가요, 배치로 대용량 데이터를 처리하는 배치 시스템인가요? 그리고 요청이 많을 때 자동으로 확장되어야 하나요?"

**왜 이 질문이 중요한가:**
AI 서비스는 일반 서비스보다 응답 시간이 길고(수 초), 비용이 요청당 발생합니다. 오토스케일링 없이 트래픽이 몰리면 비용이 순식간에 폭증할 수 있습니다.

**관련 Layer:** HOW

**배포 패턴:**

| 배포 형태 | 아키텍처 | 비용 구조 | 확장 방식 |
|---|---|---|---|
| 동기 API | FastAPI + 로드밸런서 | 서버 고정 + API 비용 | 수평 확장 |
| 비동기 큐 | Celery + Redis | 서버 + API 비용 | 워커 수 조정 |
| 서버리스 | Lambda / Cloud Functions | API 비용만 (거의) | 자동 확장 |
| 스트리밍 응답 | FastAPI SSE / WebSocket | 서버 + API 비용 | 연결 수 기반 |
| 배치 처리 | Celery Beat / 크론잡 | 서버 최소 + API 비용 | 배치 크기 조정 |

### AE-Q8: 가드레일 및 안전장치

**질문:**
> "AI 출력에 대한 안전장치가 필요한가요? 사용자가 악의적인 프롬프트를 주입(Prompt Injection)하거나, 민감한 정보를 출력하거나, 부적절한 내용을 생성하는 것을 방지해야 하나요? 규정 준수(개인정보보호법, 의료법 등) 요건이 있나요?"

**왜 이 질문이 중요한가:**
가드레일 없는 LLM 서비스는 보안 취약점이 됩니다. Prompt Injection 공격으로 시스템 프롬프트가 노출되거나, 개인정보가 출력에 포함될 수 있습니다.

**관련 Layer:** HOW

**가드레일 유형:**

| 위협 | 방어 방법 | 구현 |
|---|---|---|
| Prompt Injection | 시스템 프롬프트 분리, 입력 검증 | 입력 전처리 레이어 |
| 개인정보 출력 | PII 감지 + 마스킹 | Presidio 또는 정규식 |
| 유해 콘텐츠 | 콘텐츠 모더레이션 API | OpenAI Moderation / Azure Content Safety |
| 과도한 사용 | 사용자별 Rate Limiting | Redis 기반 카운터 |
| 할루시네이션 | RAG + 출처 표시 + 신뢰도 점수 | 소스 추적 레이어 |
| 비용 폭증 | 토큰 한도 + 비용 알림 | 프롬프트 길이 제한 + CloudWatch |

## 도메인 특화 리스크

| 리스크 | 수준 | 설명 |
|---|---|---|
| LLM 할루시네이션 | 🔴 높음 | AI가 사실처럼 보이는 거짓 정보를 생성. 사실 검증 레이어 필수 |
| API 비용 폭증 | 🔴 높음 | 프롬프트가 길거나 사용자가 많아지면 비용이 예상의 10배 이상 나올 수 있음 |
| Prompt Injection | 🟡 중간 | 악의적 사용자가 시스템 프롬프트를 우회하거나 노출시키는 공격 |
| 벤더 락인 | 🟡 중간 | 특정 LLM API에 의존하면 API 변경/가격 인상 시 대응 어려움. 추상화 레이어 설계 권장 |
| 모델 성능 저하 (Drift) | 🟡 중간 | 모델 업데이트로 기존에 잘 되던 프롬프트가 갑자기 달라질 수 있음 |
| 개인정보 처리 위반 | 🟡 중간 | 사용자 데이터를 상용 API에 보낼 때 개인정보처리방침 위반 가능성 |
| 오픈소스 모델 운영 부담 | 🟢 낮음 | GPU 서버 관리, 모델 업데이트, 모니터링 모두 직접 해야 함 |
