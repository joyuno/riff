# 기획 품질 스코어카드

AI와 함께 기획한 결과의 품질을 평가하고, 추가로 필요한 정보를 파악합니다.

---

## 평가 차원

### 1. 커버리지 (Coverage)
5개 Layer 각각의 답변 완성도를 평가합니다.

| 상태 | 기준 |
|---|---|
| 완전 (✅) | 해당 Layer의 핵심 질문 모두 답변 + 구체적 수치/예시 포함 |
| 부분 (⚠️) | 핵심 질문은 답변했으나 모호하거나 수치 없음 |
| 미완 (❌) | 해당 Layer 질문에 답변 없거나 "모르겠다"로 마무리 |

### 2. 일관성 (Consistency)
답변 간 모순을 자동 감지합니다.

**감지 패턴 목록:**

| 모순 패턴 | 감지 조건 | 경고 메시지 |
|---|---|---|
| 속도 vs 기능 충돌 | Q7="2주 이내" + Q6 반드시 목록 3개 이상 복잡 기능 | "출시 일정과 기능 범위가 충돌합니다. 둘 중 하나를 조정해야 합니다." |
| 규모 vs 단순성 충돌 | Q5=C(10,000명+) + Q8="팀 없음/혼자" | "10,000명 규모 시스템을 혼자 운영하기 어렵습니다. 관리형 서비스 사용을 강력 권장합니다." |
| 비용 vs 품질 충돌 | `reliability_critical` 태그 + Q7="2주 이내" | "신뢰성이 핵심인 프로젝트에 2주 출시는 위험합니다. QA 시간을 확보해야 합니다." |
| 측정 기준 모호 | Q10 답변에 수치 없음 | "성공 기준에 구체적인 숫자가 없습니다. 측정 불가능한 목표는 달성 여부를 알 수 없습니다." |
| 사용자 vs 기능 미스매치 | Q3="비기술자" + Q6 반드시에 "API", "CLI", "코드" 포함 | "비기술 사용자에게 기술 인터페이스를 제공하는 것은 채택률을 낮춥니다." |

### 3. 구체성 (Specificity)
답변이 아키텍처 결정을 내리기에 충분한 정보를 포함하는지 평가합니다.

| 항목 | 구체적 답변 기준 |
|---|---|
| 사용자 규모 | 숫자 범위 제시 (예: "100명~500명") |
| 출시 타임라인 | 특정 날짜 또는 기간 (예: "5월 말") |
| 성공 기준 | 측정 가능한 수치 포함 (예: "처리 시간 70% 감소") |
| 기능 우선순위 | 반드시/좋음/나중에 각 항목 명시 |
| 기술 스택 | 최소 1개 이상의 구체적 기술 언급 |

### 4. 리스크 감지 (Risk Detection)
인터뷰 전체에서 감지된 위험 요소를 분류합니다.

**높음 (🔴):** 즉시 해결하지 않으면 프로젝트 실패 가능성 높음
**중간 (🟡):** 계획 단계에서 대비책을 마련해야 함
**낮음 (🟢):** 인지하고 있으면 충분

## 스코어카드 출력 형식

```markdown
## 기획 품질 스코어카드

### 커버리지
| Layer | 프레임워크 | 상태 | 비고 |
|---|---|---|---|
| Layer 1: WHY | JTBD | ✅ 완전 | Job: "수작업 오류 줄이기" — reliability_critical 태깅 |
| Layer 2: WHO | Mom Test | ⚠️ 부분 | 실제 행동은 파악, 규모 답변 모호 ("많이 쓸 것 같아요" ← 가설적) |
| Layer 3: WHAT | Constraint | ✅ 완전 | 핵심 1개 확정 + MVP 기능 3개, 타임라인 6주 |
| Layer 4: HOW | 소크라테스 | ⚠️ 부분 | 트레이드오프 1개 탐색, 스택 미정 |
| Layer 5: MEASURE | Pre-mortem | ❌ 미완 | 실패 시나리오 미탐색 ("잘 되면 성공이죠") |

**커버리지 점수: 2/5 Layer 완전**

### 프레임워크 적용 검사
| 규칙 | 상태 |
|---|---|
| JTBD: Job이 기능이 아닌 행동으로 정의됨 | ✅ |
| Mom Test: 가설적 답변 리디렉션 수행 | ⚠️ Q5에서 미수행 |
| Constraint: "하나만" 질문으로 핵심 확인 | ✅ |
| 소크라테스: 트레이드오프에 비용 명시 | ⚠️ 1건만 수행 |
| Pre-mortem: 실패 시나리오 도출 | ❌ 미수행 |

### 일관성 검사
| 모순 유형 | 심각도 | 내용 |
|---|---|---|
| 측정 기준 모호 | 🟡 중간 | Pre-mortem 미수행으로 성공 기준 부재 |
| Mom Test 위반 | 🟡 중간 | Q5 답변이 가설적 ("많이 쓸 것 같아요") |

**일관성 이슈: 2개**

### 구체성 체크
| 항목 | 상태 |
|---|---|
| 핵심 Job (JTBD) | ✅ "수작업 오류 줄이기" |
| 실제 행동 (Mom Test) | ✅ "매주 엑셀 복붙 3시간" |
| 사용자 규모 | ❌ 모호 ("많이") — Mom Test 재질문 필요 |
| 핵심 한 가지 (Constraint) | ✅ "주문 확인 화면" |
| 출시 타임라인 | ✅ "6주 이내" |
| 실패 시나리오 (Pre-mortem) | ❌ 없음 |
| 성공 수치 | ❌ 없음 |
| 기술 트레이드오프 | ⚠️ 1건만 결정 |

**구체성 점수: 4/8 항목 완전**

### 리스크 감지
| 리스크 | 수준 | 출처 | 대응 방안 |
|---|---|---|---|
| 실패 시나리오 미탐색 | 🔴 높음 | Pre-mortem 미수행 | Layer 5 재진행 필수 |
| 사용자 규모 불명확 | 🟡 중간 | Mom Test 위반 | "실제로 사용할 사람 몇 명인가요?" 재질문 |
| 기술 스택 미확정 | 🟡 중간 | 소크라테스 미완 | 트레이드오프 추가 탐색 |

### 종합 판정
**기획 완성도: 50% — 추가 기획 필요**

미완 항목:
- [ ] Layer 2: 사용자 규모 Mom Test 재질문 (Q5 — "실제로 몇 명이 쓰고 있나요?")
- [ ] Layer 5: Pre-mortem 전체 수행 (Q10~Q10-리스크)

완료 후 아키텍처 결정 + QA 시나리오 + 초기 항체 자동 생성됩니다.
```

## 자동 판정 기준

### 즉시 진행 가능 (완성도 80% 이상)
- 5개 Layer 중 4개 이상 완전
- 일관성 이슈 없음
- 리스크 높음 항목 없음
- → 바로 `answer-to-arch.md` 매핑 실행

### 부분 진행 (완성도 60-79%)
- 5개 Layer 중 3개 완전
- 일관성 이슈 1개 이하
- → 미완 항목 재질문 후 진행, 또는 사용자 동의 하에 가정(assumption)을 명시하고 진행

### 재진행 필요 (완성도 59% 이하)
- Layer 1 또는 Layer 3 미완 (핵심 레이어)
- 리스크 높음 항목 2개 이상
- 일관성 이슈 2개 이상
- → 인터뷰 부분 재진행 필요, 미완 Layer부터 재시작

## 가정(Assumption) 처리

사용자가 모르거나 미정인 항목에 대해 가정을 사용할 때는 반드시 명시합니다:

```markdown
### 적용된 가정
| 항목 | 가정 값 | 근거 | 재검토 시점 |
|---|---|---|---|
| 동시 사용자 수 | 100~500명 (선택지 B) | 팀 내부 도구 + 중소기업 규모 | MVP 출시 후 1개월 실측 |
| 출시 타임라인 | 6주 | "빠른 출시" 언급 기반 | 킥오프 미팅에서 확정 |
```

가정이 있는 채로 진행할 경우, 최종 아키텍처 문서에도 동일하게 표시합니다.
