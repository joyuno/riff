# 인터뷰 종료 조건 엔진

## 현재 문제

현재 riff-interview는 10개 고정 질문을 순서대로 진행한다. 다음 문제가 있다:
- "정보가 충분한지" 판단하는 로직 없음
- 사용자가 이미 답한 내용을 중복 질문할 수 있음
- 도메인에 따라 필요한 질문 수가 다른데 고정
- 사용자가 긴 답변에서 여러 Layer를 한번에 답해도 감지 못함

## 해결: 스코어카드 기반 종료 조건

### 1. 충분성 스코어카드

매 답변 후 각 Layer의 충분성을 0~100%로 계산한다.

```
  Layer 1 (WHY):    ██████████ 100%   ← 답변 완료
  Layer 2 (WHO):    ████████░░  80%   ← 부분 답변
  Layer 3 (WHAT):   ██████░░░░  60%   ← 부분 답변
  Layer 4 (HOW):    ░░░░░░░░░░   0%   ← 미수집
  Layer 5 (MEASURE):░░░░░░░░░░   0%   ← 미수집

  종합 충분성: 48%
  임계값: 70% → 아직 부족, 질문 계속
```

### 2. 필수 태그 시스템

각 Layer에 "필수 태그"를 정의한다. 답변에서 태그를 자동 추출하여 충족 여부를 판단한다.

**Layer 1 (WHY) 필수 태그:**
- `problem`: 해결하려는 핵심 문제
- `motivation`: 왜 지금 만드는지
- `impact`: 안 만들면 벌어지는 일

**Layer 2 (WHO) 필수 태그:**
- `user_type`: 핵심 사용자 유형
- `user_count`: 동시 사용자 규모
- `pain_point`: 핵심 불편함

**Layer 3 (WHAT) 필수 태그:**
- `must_have`: MVP 필수 기능 목록
- `nice_to_have`: 후순위 기능
- `timeline`: MVP 목표 시점

**Layer 4 (HOW) 필수 태그:**
- `stack`: 기술 스택 선호
- `constraints`: 기술적 제약
- `tradeoff_decision`: 주요 트레이드오프 결정

**Layer 5 (MEASURE) 필수 태그:**
- `success_metric`: 성공 측정 기준
- `measurement_method`: 측정 방법

### 3. 충분성 계산 방법

```
Layer 충분성 = (충족된 필수 태그 수) / (총 필수 태그 수) × 100%

종합 충분성 = Layer별 충분성의 가중 평균
  WHY:     가중치 25% (방향 결정의 핵심)
  WHO:     가중치 25% (아키텍처 규모 결정)
  WHAT:    가중치 20% (범위 결정)
  HOW:     가중치 15% (기술 결정, 후순위 가능)
  MEASURE: 가중치 15% (후반에 확정 가능)
```

### 4. 답변에서 멀티 태그 추출

사용자가 한 답변에서 여러 Layer의 태그를 동시에 채울 수 있다:

```
사용자 답변:
  "소규모 쇼핑몰 운영자가 100명 정도 쓸 건데,
   주문 처리가 너무 느려서 자동화가 필요해"

추출:
  ✅ WHO - user_type: "쇼핑몰 운영자"
  ✅ WHO - user_count: "~100"
  ✅ WHY - problem: "주문 처리 느림"
  ✅ WHY - motivation: "자동화 필요"

→ Layer 1 충분성: 67% (2/3)
→ Layer 2 충분성: 67% (2/3)
→ Q1(WHY 질문), Q5(사용자 수 질문) 건너뜀
```

### 5. 중복 질문 방지

다음 질문을 선택할 때:
1. 충분성이 가장 낮은 Layer를 우선 선택
2. 해당 Layer에서 미충족 태그에 해당하는 질문만 선택
3. 이미 답변된 태그의 질문은 건너뜀
4. 도메인 특화 질문이 있으면 범용 질문보다 우선

### 6. 종료 조건 (OR 조건 — 하나라도 충족 시 종료)

| 조건 | 기준 | 이유 |
|------|------|------|
| 충분성 달성 | 종합 충분성 ≥ 70% | 핵심 정보 수집 완료 |
| 사용자 명시적 종료 | "시작해", "충분해", "그만" 발화 | 사용자 의사 존중 |
| 최대 질문 수 도달 | 질문 예산별 상한 (아래 표) | 사용자 인내심 보호 |
| 정보 포화 | 연속 2개 질문에서 새 태그 0개 | 더 물어도 의미 없음 |

### 7. 질문 예산과 최대 제한 연동

| 질문 예산 | 최대 질문 수 | 최소 충분성 | 미수집 처리 |
|----------|------------|-----------|-----------|
| A (매 단계 확인) | 12개 | 80% | 부족한 것 추가 질문 |
| B (핵심만) | 8개 | 70% | 합리적 기본값 제안 |
| C (최소한) | 5개 | 50% | 기본값 적극 활용 |
| D (전자동) | 2개 | 30% | WHY+WHO만, 나머지 전부 기본값 |

### 8. 미수집 정보 처리

최대 제한에 도달했거나 사용자가 종료를 요청하면:
- 미수집 태그를 "합리적 기본값"으로 채움
- 기본값은 답변→아키텍처 매핑 테이블(answer-to-arch.md)의 가장 일반적인 값 사용
- 사용자에게 가정 목록을 제시: "다음 가정으로 진행합니다: [목록]. 수정할 것이 있으면 말씀해주세요."

### 9. 전체 로직 플로우

```
인터뷰 시작
  │
  ▼
┌─────────────────────────────┐
│ 다음 질문 선택              │
│                              │
│ 1. 충분성 가장 낮은 Layer    │
│ 2. 해당 Layer의 미충족 태그  │
│ 3. 이미 답변된 태그 건너뜀   │
│ 4. 도메인 파일 있으면 특화질문│
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 질문 → 사용자 답변           │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 답변에서 태그 자동 추출      │
│ (여러 Layer 동시 채움 가능)  │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 스코어카드 업데이트          │
│ 각 Layer 충분성 재계산       │
└──────────┬──────────────────┘
           │
           ▼
┌─────────────────────────────┐
│ 종료 조건 체크              │
│                              │
│ □ 충분성 ≥ 70%?             │
│ □ 질문 수 ≥ 최대 제한?      │
│ □ 사용자가 "시작해"?         │
│ □ 연속 2개 새 정보 없음?     │
│                              │
│ 하나라도 ✅ → 인터뷰 종료    │
│ 전부 □   → 다음 질문으로     │
└──────────┬──────────────────┘
           │
           ▼ (종료 시)
┌─────────────────────────────┐
│ 미수집 태그 → 기본값 채움    │
│ 스코어카드 최종 출력         │
│ 가정 목록 사용자에게 제시    │
│ → BUILD 단계로 진행         │
└─────────────────────────────┘
```

### 10. 스코어카드 출력 형식

```markdown
## 인터뷰 완료 — 스코어카드

| Layer | 충분성 | 수집된 태그 | 미수집 (기본값) |
|-------|--------|-----------|---------------|
| WHY | 100% | problem, motivation, impact | — |
| WHO | 67% | user_type, user_count | ⚠️ pain_point (기본: "수동 작업 비효율") |
| WHAT | 100% | must_have, nice_to_have, timeline | — |
| HOW | 33% | stack | ⚠️ constraints, tradeoff (기본: 표준 웹) |
| MEASURE | 0% | — | ⚠️ 전부 기본값 (기본: "기능 동작 확인") |

**종합 충분성: 68%**
**질문 수: 6/8 (예산 B)**
**가정 3개로 진행합니다. 수정할 것이 있으면 말씀해주세요.**
```
