## 도입 멘트

"저는 데이터 엔지니어입니다. 데이터가 어떻게 흐르고 저장되는지 설계해볼게요."

---

## Q1: 데이터 볼륨 예측

"서비스 시작 후 1년 뒤 예상 데이터 규모는?"

선택지:
- A: 소규모 (사용자 1만 명 미만, 데이터 1GB 미만)
- B: 중규모 (사용자 10만 명, 데이터 10~100GB)
- C: 대규모 (사용자 100만 명 이상, 데이터 1TB 이상)
- D: 모름 (가장 보수적 추정으로 진행)

태깅: A→`data_small`, B→`data_medium`, C→`data_large`

---

## Q2: 실시간 vs 배치 처리

"데이터 처리 방식은?"

선택지:
- A: 실시간 처리만 (사용자 요청마다 즉시 처리)
- B: 배치 처리만 (밤마다 일괄 처리, 리포트 등)
- C: 혼합 (실시간 + 야간 배치)
- D: 이벤트 드리븐 (특정 이벤트 발생 시 트리거)

실시간 선택 시: "응답 시간 허용 범위는? (< 100ms / < 1s / < 5s)"
배치 선택 시: "처리 주기는? (1시간 / 하루 / 주간)"

---

## Q3: 컨테이너 / 오케스트레이션

"인프라 관리 방식은?"

선택지:
- A: 단순 서버 (VPS, 단일 서버로 시작)
- B: Docker Compose (멀티 서비스, 로컬 개발 친화)
- C: Kubernetes (k8s) — 대규모 자동 스케일링
- D: 관리형 서비스 (AWS ECS, Cloud Run, Railway)

**K8s 추천 기준**:
- `data_large` 태그 AND 동시 접속 1,000+ → K8s 추천
- `data_small` OR `data_medium` → Docker Compose 또는 관리형 서비스 추천

질문: "[추천 기준 기반 추천]으로 시작할까요?"

---

## Q4: 로그 / 모니터링

"운영 중 시스템을 어떻게 관찰할까요?"

선택지:
- A: 기본 (서버 로그만, 무료)
- B: 중급 (Sentry 에러 추적 + 기본 메트릭)
- C: 고급 (Prometheus + Grafana + 분산 트레이싱)
- D: 관리형 (DataDog, New Relic)

---

## Q5: 데이터 이동 최적화

"다음 최적화 중 필요한 것은?"

- [ ] CDN (정적 파일, 이미지 글로벌 배포)
- [ ] 캐싱 레이어 (Redis — 자주 읽는 데이터)
- [ ] 검색 엔진 (Elasticsearch, Meilisearch — 전문 검색)
- [ ] 메시지 큐 (Kafka, RabbitMQ — 비동기 처리)
- [ ] 데이터 웨어하우스 (BigQuery, Redshift — 분석용)
- [ ] 없음 (DB 직접 쿼리로 충분)
