# Pulse 질문 엔진 — AI와 함께하는 상세 기획

AI는 코드를 잘 짭니다. 하지만 **"무엇을 만들어야 하는가"는 당신의 머릿속에만 있습니다.**

이 스킬은 검증된 질문 프레임워크 4가지를 결합해서, 당신의 아이디어를 AI가 정확히 구현할 수 있는 수준의 상세 기획으로 바꿔줍니다. 개발 경험이 없어도 됩니다. 올바른 질문에 답하기만 하면, AI가 나머지를 합니다.

---

## 언제 이 스킬을 사용하나

다음 키워드나 의도가 감지되면 이 스킬을 활성화합니다:

- "프로젝트 설계해줘", "요구사항 정리", "기획이 필요해"
- "뭘 만들어야 할지 모르겠어", "어떻게 시작해야 해"
- "스펙 정리해줘", "아이디어가 있는데 구체화하고 싶어"
- "MVP 뭐부터 해야 해", "어떤 기술 쓸지 모르겠어"
- Pulse의 ASK 단계 진입 시 자동 호출

---

## 4가지 질문 프레임워크

Pulse는 단일 방법론이 아닌, **상황에 맞는 최적의 질문법**을 사용합니다.

| 프레임워크 | 핵심 원리 | 강점 | 적용 Layer |
|---|---|---|---|
| **JTBD** (Jobs-to-Be-Done) | 사용자가 이 제품을 "고용"해서 해결할 일은? | scope를 자연스럽게 줄임 | Layer 1: WHY |
| **Mom Test** | "쓸 건가요?" 대신 "지금 어떻게 하고 계세요?" | 가설이 아닌 실제 행동 기반 | Layer 2: WHO |
| **Constraint Forcing** | "1주밖에 없다면 뭘 만들까?" | 발산→수렴 강제 | Layer 3: WHAT |
| **Pre-mortem** | "이게 실패했다면, 왜 실패했을까?" | 리스크 사전 제거 | Layer 5: MEASURE |

Layer 4(HOW)에서는 소크라테스식 대화로 트레이드오프를 탐색합니다.

> **핵심 철학:** 질문은 답을 얻기 위한 것이 아닙니다. 당신이 이미 알고 있지만 아직 정리하지 못한 것을 꺼내는 과정입니다.

---

## 5-Layer 기획 프레임

인터뷰는 5개의 레이어로 구성되며, 각 레이어의 답변이 다음 레이어의 질문을 결정합니다. 상세 질문 트리는 `references/layers.md`를 참조합니다.

### Layer 1: WHY — 이 제품이 해결할 "일" (JTBD)
> "사용자가 이 제품을 '고용'해서 시킬 일이 정확히 뭔가?"

기능이 아닌 **Job**에서 시작합니다. "쇼핑몰을 만들고 싶어요"가 아니라 "소상공인이 주문을 놓치지 않고 처리하는 일"이 진짜 시작점입니다. Job이 명확하면 필요없는 기능이 자연스럽게 탈락합니다.

**핵심 질문:**
- Q1: "이 제품이 없으면, 그 일을 지금 어떻게 하고 있나요? 그 방식의 가장 큰 문제는?"
- Q2: "이 제품을 '고용'한 사용자가 하루가 끝날 때 '오늘 이걸 써서 다행이다'라고 느끼는 순간은 어떤 순간인가요?"

**자동 태깅:**
- 기존 방식이 느리다 → `performance_critical`
- 수동 실수가 많다 → `reliability_critical`
- 새 제품/시장 개척 → `market_risk`

---

### Layer 2: WHO — 실제 행동 기반 사용자 파악 (Mom Test)
> "그 사람이 지금 실제로 어떻게 하고 있나?"

"사용하실 건가요?" 같은 가설적 질문은 하지 않습니다. Mom Test 원칙에 따라 **실제 행동과 과거 경험**만 물어봅니다. 엄마한테 "내 앱 쓸 거야?"라고 물으면 무조건 "쓸게"라고 하지만, "지난주에 그 문제를 어떻게 해결했어?"라고 물으면 진짜 답이 나옵니다.

**핵심 질문:**
- Q3: "이 문제를 겪는 사람을 한 명 떠올려보세요. 그 사람이 지난주에 이 문제를 어떻게 해결했나요? 구체적인 상황으로요."
- Q4: "그 과정에서 가장 짜증났던 순간은 언제였나요? 돈이나 시간을 얼마나 낭비했나요?"
- Q5: "이 서비스를 동시에 쓸 사람이 몇 명 정도 될까요?"
  - 선택지 A: 1~10명 (팀 내부 도구)
  - 선택지 B: 100~1,000명 (중소 서비스)
  - 선택지 C: 10,000명 이상 (공개 서비스/스케일 필요)

**Mom Test 경고 규칙:**
- 사용자가 "~할 것 같아요", "~하면 좋겠어요" 같은 가설적 답변을 하면 → "실제로 그런 적이 있었나요? 구체적인 상황을 알려주세요." 로 리디렉션
- "다 필요해요"라고 하면 → "지난주에 가장 급했던 한 가지만 고르면요?"로 수렴

---

### Layer 3: WHAT — 제약 조건으로 핵심 찾기 (Constraint Forcing)
> "만약 1주일밖에 없다면, 딱 하나만 만든다면 뭘 만들까?"

기능 욕심은 프로젝트를 죽입니다. Constraint Forcing은 인위적인 제약을 걸어서 **진짜 핵심이 무엇인지** 강제로 드러나게 합니다. "다 중요해요"는 아무것도 중요하지 않다는 뜻입니다.

**핵심 질문:**
- Q6: "극단적으로 단순화해볼게요. 이 서비스에서 딱 하나의 화면, 하나의 버튼만 있다면 그게 뭔가요? 사용자가 그 버튼을 누르면 어떤 일이 일어나나요?"
- Q6-확장: "좋아요, 그게 핵심이네요. 이제 거기에 2~3개를 더 추가한다면? '반드시 있어야 함' / '있으면 좋음' / '나중에'로 분류해주세요. 각각 최대 3개씩."
- Q7: "첫 버전을 언제까지 써보고 싶으세요?"
  - 선택지 A: 2주 이내 → 에이전트 수 최대화, 병렬화 극대화
  - 선택지 B: 1~2개월 → 표준 팀 구성
  - 선택지 C: 3개월 이상 → 품질/테스트 강화 전략

**Constraint Forcing 원칙:**
- 처음에는 극단적으로 좁히고, 점진적으로 확장합니다
- "하나만"에서 시작해서 "세 개까지"로 넓히면 우선순위가 자연스럽게 정해집니다
- Q7의 시간 제약이 Q6의 범위를 현실적으로 조정합니다

---

### Layer 4: HOW — 트레이드오프 탐색 (소크라테스)
> "A와 B 중 어느 쪽이고, 그 선택의 비용은?"

기술 결정은 비즈니스 결정입니다. 소크라테스 대화법으로 **트레이드오프를 가시화**하고, 사용자가 스스로 결정하게 합니다. AI가 "이걸 쓰세요"라고 정하는 것이 아니라, 선택의 결과를 이해한 상태에서 사용자가 결정합니다.

**핵심 질문:**
- Q8: "기술에 대해 잘 모르셔도 괜찮습니다. 한 가지만 알려주세요 — 이미 사용하고 있거나 익숙한 도구가 있나요? (엑셀, 노션, 특정 앱 등 뭐든)"
- Q9: [Layer 1 태깅에 따라 동적으로 생성 — 소크라테스식 대화]
  - `performance_critical` 태그 → "속도가 중요하다고 하셨죠. 두 가지 방향이 있어요: '각 사용자에게 빠르게 응답하는 것'과 '많은 사용자를 동시에 처리하는 것'. 어떤 상황이 더 가까운가요?"
  - `reliability_critical` 태그 → "실수가 문제라고 하셨죠. '실수를 아예 못하게 막는 것'과 '실수하더라도 바로 되돌릴 수 있는 것' 중 어느 쪽이 더 현실적인가요?"
  - `market_risk` 태그 → "새로운 시장이니 빠르게 테스트하는 게 중요하겠네요. '2주 안에 출시하지만 나중에 구조를 바꿔야 할 수도 있는 것'과 '한 달 걸리지만 확장하기 쉬운 구조' 중 어느 쪽이 맞나요?"

**소크라테스 대화 원칙:**
- 기술 용어 대신 **결과**로 설명합니다 ("React vs Vue" 대신 "빠른 출시 vs 유연한 구조")
- 모든 선택지에 **비용**을 붙입니다 ("빠르지만 나중에 바꿔야 할 수도 있어요")
- 비개발자가 이해할 수 있는 비유를 사용합니다

> Layer 4는 소크라테스 기본 질문 이후 전문가 기술 결정으로 확장된다. 이전 Layer 태그에 따라 UI·인프라·DB·AI 질문이 자동 선택된다. 상세: `references/enriched-layers.md`

---

### Layer 5: MEASURE — 실패 시나리오 사전 제거 (Pre-mortem)
> "3개월 후 이 프로젝트가 실패했다고 칩시다. 왜 실패했을까요?"

전통적인 "성공 기준을 정해주세요"는 추상적인 답만 나옵니다. Pre-mortem은 반대로 **실패를 먼저 상상하게** 해서 구체적인 리스크와 측정 기준을 동시에 끌어냅니다.

**핵심 질문:**
- Q10: "한번 상상해볼게요. 3개월 후, 이 서비스를 만들었는데 아무도 안 쓰거나 실패했어요. 가장 가능성 높은 이유가 뭘까요?"
- Q10-후속: "그러면 그 실패를 막으려면, 출시 첫 달에 어떤 숫자를 봐야 '잘 되고 있다'고 확신할 수 있을까요?"
  - 선택지 A: 사용량 지표 (DAU, 요청 수, 처리 건수)
  - 선택지 B: 비즈니스 지표 (매출, 전환율, 고객 유지율)
  - 선택지 C: 품질 지표 (오류율, 응답시간, 정확도)
- Q10-리스크: "방금 말씀하신 실패 시나리오를 막기 위해, MVP에서 반드시 검증해야 할 가설은 뭘까요?"

**Pre-mortem 원칙:**
- 실패를 상상하면 성공 기준보다 훨씬 구체적인 답이 나옵니다
- Q10의 실패 시나리오는 QA 전략과 직결됩니다 (Pulse QA가 자동으로 해당 시나리오를 테스트)
- Pulse Immunity(면역 시스템)의 초기 항체로 등록됩니다

## 질문 설계 원칙

모든 질문은 다음 기준을 충족해야 합니다:

1. **비개발자가 5초 안에 이해할 수 있는 질문**
   - 나쁜 예: "SSR과 CSR 중 어느 방식을 선호하시나요?"
   - 좋은 예: "첫 화면이 바로 보이는 게 중요한가요, 아니면 한 번 로딩 후 빠르게 전환되는 게 중요한가요?"

2. **"예/아니오"가 아닌 "A vs B" 또는 경험 기반 질문**
   - 나쁜 예: "실시간 기능이 필요한가요?"
   - 좋은 예: "알림이 1분 늦게 와도 괜찮나요, 아니면 즉시 와야 하나요? 실제로 1분 늦게 오면 어떤 문제가 생기나요?"

3. **트레이드오프를 비용과 함께 노출하는 질문**
   - 모든 선택에는 비용이 있습니다. 질문은 그 비용을 일상 언어로 명시해야 합니다.
   - 예: "빠르게 만들면 2주 안에 써볼 수 있지만, 사용자가 1000명 넘으면 다시 만들어야 할 수 있어요. 괜찮으신가요?"

4. **과거 경험 > 미래 의견**
   - Mom Test 원칙: 미래에 대한 의견("쓸 것 같아요")보다 과거 행동("지난주에 어떻게 했나요")이 100배 정확합니다.

5. **각 질문에 "왜 이걸 묻는지" 설명 포함**
   - 사용자가 질문의 의도를 이해할 때 더 정확한 답변을 줍니다.
   - 예: "이 질문은 서버 구조를 결정하기 위한 거예요. 답변에 따라 비용이 월 $5에서 $500까지 달라질 수 있어요."

## 도메인 전문가 모드

인터뷰 시작 시 사용자의 프로젝트 설명에서 도메인을 자동 감지합니다.

### 감지 → 로드 매핑

| 감지 키워드 | 로드할 파일 |
|---|---|
| 웹, React, Next.js, Vue, 웹사이트, 랜딩페이지 | `domains/web-development.md` |
| 스마트스토어, 드롭쉬핑, 쿠팡, 네이버쇼핑, 셀러, 도매 | `domains/smart-store.md` |
| 유튜브, 영상, 릴스, 숏폼, 대본, 편집, 썸네일 | `domains/video-creation.md` |
| 퀀트, 자동매매, 트레이딩, 백테스트, 알고리즘매매 | `domains/quant-trading.md` |
| AI, LLM, GPT, Claude, RAG, 파인튜닝, 에이전트 | `domains/ai-engineering.md` |

### 폴백 전략

도메인이 감지되지 않으면:
1. Layer 2 Q3에서 사용자의 실제 상황을 파악한 뒤 도메인을 재감지합니다.
2. 그래도 매칭되지 않고 UI/화면 단어가 1개 이상 등장하면 → **`web-development`를 기본 도메인으로 가정**하고 진행 (Pulse가 처음 만들 때 가장 흔한 경우).
3. 그 외 매칭이 전혀 없으면 5-Layer 범용 질문으로 진행.
4. 인터뷰 완료 후 "이 도메인에 특화된 추가 질문이 필요하신가요?" 안내를 제공합니다.

## 인터뷰 진행 방식

### 단계별 흐름

```
1. 도입 — 함께 기획하기 (1분)
   → "아이디어를 함께 구체화해볼게요. 제가 몇 가지 질문을 드릴 텐데,
      정답은 없고 솔직한 생각이면 충분합니다."
   → 프로젝트 한 줄 소개 요청
   → 도메인 감지 및 전문가 질문 로드

2. Layer 1: WHY — JTBD (3분)
   → Q1~Q2: 이 제품이 해결할 "일"이 뭔지 파악
   → 각 답변에서 태그 자동 추출
   → "좋아요, 핵심 문제가 [요약]이네요. 이제 이걸 겪는 사람에 대해 더 알아볼게요."

3. Layer 2: WHO — Mom Test (5분)
   → Q3~Q5: 실제 행동과 고통을 구체화
   → 가설적 답변 감지 시 리디렉션
   → "이제 무엇을 만들지 좁혀볼게요."

4. Layer 3: WHAT — Constraint Forcing (3분)
   → Q6~Q7: 극단적 단순화 → 점진적 확장
   → 기능 목록 확정 및 우선순위 정렬
   → "핵심이 잡혔어요. 이제 어떻게 만들지 같이 정할게요."

5. Layer 4: HOW — 소크라테스 대화 (5분)
   → Q8~Q9: 트레이드오프를 비개발자 언어로 탐색
   → 도메인 전문가 질문 추가 (해당하는 경우)
   → "거의 다 왔어요. 마지막으로 리스크를 점검해볼게요."

6. Layer 5: MEASURE — Pre-mortem (3분)
   → Q10: 실패 시나리오 상상 → 성공 기준 역추출
   → 실패 시나리오 → 초기 항체 등록 (Immunity 연동)

7. 기획 완성 — 결과 출력
   → 스코어카드 출력 (references/scorecard.md)
   → 아키텍처 매핑 출력 (references/answer-to-arch.md)
   → 실패 시나리오 → QA 테스트 케이스 자동 생성
   → "기획이 완성됐어요. 이제 만들어볼까요?"

8. Layer 4: HOW 기술 결정 라운드테이블 (15분)
   → HOW-A: UI 디자이너 (디자인 톤, 스킬 선택)
   → HOW-B: 자료 리서처 (GitHub 리서치, 스킬 설치)
   → HOW-C: 프론트엔드 (화면 목록, 라우팅)
   → HOW-D: 백엔드 (스택, 인증, 외부 연동)
   → HOW-E: 데이터 엔지니어 (인프라, 캐싱)
   → HOW-F: DB 설계자 (DB 스택, 스키마)
   → HOW-G: AI 엔지니어 (ai_required/optional 태그 있을 때만)
   → 이전 Layer 태그에 따라 관련 없는 전문가 자동 건너뜀
   → 상세: references/enriched-layers.md 참조

9. Layer 5: MEASURE + Prompt Engineer (5분)
   → Pre-mortem 기본 질문
   → Prompt Engineer: 충돌 감지 + P0/P1/P2 최종 확인
   → master-plan.md 생성
   → "기획 완료. BUILD를 시작할까요?"
```

### 진행 규칙

- 한 번에 질문 하나만 합니다. 여러 질문을 한꺼번에 던지지 않습니다.
- **비개발자 우선 언어:** 기술 용어가 필요하면 반드시 일상 비유를 함께 씁니다.
- 답변이 모호하면 프레임워크에 맞는 리디렉션을 합니다:
  - 가설적 답변 → Mom Test 리디렉션: "실제로 그런 적이 있었나요?"
  - "다 필요해요" → Constraint Forcing: "딱 하나만 고르면요?"
  - 추상적 성공 기준 → Pre-mortem: "실패한다면 이유가 뭘까요?"
- Layer 간 이동 시 **요약 + 자연스러운 전환**을 합니다.
- 종료 조건은 시간이 아닌 **스코어카드 기반**입니다. 상세 기준은 `references/termination-engine.md` 참조.
  - 기본: 종합 충분성 ≥ 70% 또는 최대 질문 수 도달 시 종료
  - 사용자가 "시작해", "충분해", "그만" 발화 시 즉시 종료
  - 연속 2개 질문에서 새 태그 없으면 정보 포화로 종료
- 전체 톤은 **"AI가 심문한다"가 아닌 "AI와 함께 기획한다"**입니다.

```markdown
- 핵심 고통: [Q4 답변]
- 규모: [Q5 선택지]

- 핵심 한 가지: [Q6 첫 답변 — 하나의 화면, 하나의 버튼]

```
