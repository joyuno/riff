# pulse-immunity: 실패 면역 시스템

## 면역 시스템 비유

인체의 면역계는 한번 만난 바이러스를 기억한다. 다음번에 같은 바이러스가 들어오면 즉시 항체를 투입하여 발병을 막는다.

pulse-immunity는 소프트웨어 개발에 같은 원리를 적용한다. 한번 겪은 버그(병)는 항체(예방 규칙)로 저장된다. 비슷한 상황이 다시 나타나면 항체가 자동으로 주입되어 에이전트가 같은 실수를 반복하지 않도록 막는다.

### 기존 방식의 문제

- 에이전트 A가 API 응답 래핑을 빠뜨려 버그 발생
- 수정 후 종료
- 다음 세션의 에이전트 B가 같은 실수 반복
- 또 수정 → 무한 반복

### 면역 시스템의 해결

- 에이전트 A가 버그 발생 → LEARN 단계에서 항체 생성
- 다음 세션의 에이전트 B가 API 작업 시작 → 관련 항체 자동 주입
- 에이전트 B는 체크리스트를 확인하고 실수 방지

## 항체 파일 구조

항체는 `.pulse/antibodies/` 디렉토리에 Markdown 파일로 저장된다.

```
.pulse/
└── antibodies/
    ├── api-response-wrapping.md
    ├── auth-token-expiry.md
    ├── db-n-plus-one.md
    └── ...
```

### 파일 이름 규칙

- 형식: `{영역}-{문제-요약}.md`
- 예시: `api-missing-error-handler.md`, `ui-loading-state-forgotten.md`
- 소문자, 하이픈 구분, 공백 금지

### 파일 내부 구조

파일 상단에 YAML frontmatter, 이후 섹션별 내용:

```
---
name: {항체명}
type: {boundary|logic|ui|performance|security}
severity: {critical|high|medium|low}
discovered: {YYYY-MM-DD}
recurrence: {숫자}
last_seen: {YYYY-MM-DD}
status: {active|weakened|dormant}
---

## 버그 설명
## 근본 원인
## 예방 체크리스트
## 관련 파일
## 재발 이력
```

상세 스키마는 `references/antibody-schema.md` 참조.

## 항체 생명주기

### 1단계: 버그 발견 → 새 항체 생성

LEARN 단계에서 에이전트는 다음을 수행한다:

```
1. 버그의 근본 원인 파악
2. 유사 항체가 이미 존재하는지 .pulse/antibodies/ 검색
3. 없으면 새 항체 파일 생성
4. severity 판단:
   - critical: 데이터 손실, 보안 취약점, 서비스 중단
   - high: 주요 기능 오작동
   - medium: 부분적 기능 오류
   - low: UI 불일치, 경고 메시지
5. status: active로 시작
6. recurrence: 0으로 시작
```

### 2단계: 비슷한 버그 재발 → 항체 강화

같은 유형의 버그가 다시 발생하면:

```
1. 기존 항체 파일 식별
2. recurrence 카운터 +1
3. last_seen 날짜 업데이트
4. 재발 이력 테이블에 행 추가
5. 예방 체크리스트 검토:
   - 기존 체크리스트가 왜 통하지 않았는지 분석
   - 더 구체적이고 강력한 체크 항목 추가
6. 필요 시 severity 상향
```

### 3단계: 5회 연속 재발 없음 → 항체 약화

마지막 발견일로부터 90일이 지나고 recurrence가 증가하지 않으면:

```
1. status: active → weakened
2. BUILD 단계 자동 주입에서 제외
3. 파일은 보존 (기록 목적)
4. 수동으로 다시 활성화 가능
```

약화 기준: `last_seen`으로부터 90일 경과 AND 그 기간 중 재발 없음

### 상태 전이 요약

```
[없음] ──버그 발견──→ [active]
[active] ──재발──→ [active] (강화됨)
[active] ──90일 무재발──→ [weakened]
[weakened] ──재발──→ [active] (재활성화)
[weakened] ──180일 무재발──→ [dormant]
[dormant] ──수동 활성화──→ [active]
```

## 자동 주입 메커니즘

### BUILD 단계에서의 주입

에이전트가 BUILD 단계를 시작할 때 오케스트레이터는:

```
1. 현재 작업의 파일 경로, 모듈명, 기능 키워드 수집
2. .pulse/antibodies/ 에서 status: active인 항체 목록 로드
3. 관련성 매칭 수행 (아래 기준)
4. 매칭된 항체의 예방 체크리스트를 에이전트 프롬프트에 포함
5. 에이전트는 작업 완료 전 체크리스트 확인 의무
```

### 관련성 매칭 기준

항체가 현재 작업에 관련된 것으로 판단하는 조건:

| 기준 | 설명 |
|------|------|
| 파일 경로 일치 | 항체의 관련 파일이 현재 작업 파일과 겹침 |
| 타입 일치 | 항체의 type이 현재 작업 유형과 같음 (boundary, ui 등) |
| 키워드 일치 | 항체 설명에 현재 작업 키워드 포함 (API, 인증, 결제 등) |
| severity | critical/high 항체는 타입만 일치해도 항상 주입 |

### 주입 형식

에이전트 프롬프트에 삽입되는 블록:

```
## 면역 체크리스트 (이전 버그 패턴 기반)

다음 항목은 이 유형의 작업에서 과거에 발생한 버그를 방지하기 위한 체크리스트입니다.
작업 완료 전 반드시 확인하세요.

### [api-response-wrapping] (high)
- [ ] API 응답이 { data: ... } 형태로 래핑되어 있는가?
- [ ] 에러 응답이 { error: { code, message } } 형태인가?
- [ ] 빈 배열과 null을 구분하여 처리했는가?
```

## 항체 검색

에이전트 또는 오케스트레이터가 수동으로 항체를 검색할 때:

```
검색 기준:
- 파일 경로: "src/api/users.ts"와 관련된 항체
- 유형: type: boundary인 항체 모두
- 심각도: severity: critical인 항체 모두
- 상태: status: active인 항체 모두
- 키워드: 이름/설명에 "인증"이 포함된 항체
```

## LEARN 단계 실행 절차

Pulse의 LEARN 단계에서 pulse-immunity는 다음 순서로 실행된다:

```
1. QA 결과 수신 (Tier별 실패 목록)
2. 실패 항목을 버그 유형별로 분류
3. 각 버그에 대해:
   a. 기존 항체 검색 (유사도 매칭)
   b. 기존 항체 있음: 항체 강화 (recurrence +1, 체크리스트 업데이트)
   c. 기존 항체 없음: 새 항체 생성
4. 약화 대상 항체 스캔 (90일 무재발 확인)
5. 학습 요약 보고:
   - 신규 항체 N개 생성
   - 기존 항체 M개 강화
   - 약화된 항체 K개
```

## 운영 원칙

1. **기록 우선**: 항체 파일은 삭제하지 않는다. status를 dormant로 변경할 뿐.
2. **구체성**: 체크리스트 항목은 "확인할 것"이 아닌 "무엇을 어떻게 확인"인지 명시.
3. **최소 주입**: 관련성 없는 항체는 주입하지 않는다. 노이즈는 무시된다.
4. **강화 우선**: 새 항체 생성 전 기존 항체와의 유사성을 먼저 확인.
5. **팀 공유**: .pulse/antibodies/ 는 git에 커밋하여 팀 전체가 공유.
