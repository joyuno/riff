# pulse-dna: 사용자 DNA 프로파일 시스템

## 개념

개발자마다 고유한 작업 방식이 있다. 어떤 사람은 코드 설명을 최소화하고 결과만 원한다. 어떤 사람은 선택지를 제시받기보다 직접 지시하는 것을 선호한다. 어떤 사람은 TypeScript를 좋아하고 테스트를 먼저 작성한다.

pulse-dna는 이 패턴을 세션에서 세션으로 학습하여 저장한다. 사용자가 매번 "이렇게 해줘"라고 가르치는 수고를 없애는 것이 목적이다.

### DNA의 의미

DNA는 생물의 설계도다. 변하지 않는 것이 아니라, 환경에 반응하며 발현 방식이 달라진다.
사용자 DNA도 같다. 고정된 프로파일이 아니라 새로운 관찰이 쌓일수록 더 정확해지는 살아있는 문서다.

---

## 프로파일 저장 위치

```
.pulse/
└── user-dna/
    ├── preferences.md       ← 커뮤니케이션 스타일, 설명 상세도, 보고 형식
    ├── domain-knowledge.md  ← 사용자가 잘 아는/모르는 기술 영역
    ├── pain-points.md       ← 자주 불만인 것들, 원하지 않는 패턴
    └── decision-history.md  ← 과거 선택과 이유, 반복되는 의사결정 패턴
```

프로파일 스키마 상세는 `references/profile-schema.md` 참조.

## 학습 메커니즘

### 자동 학습: 수정 요청 분석

에이전트가 결과물을 냈을 때 사용자가 수정을 요청하면, LEARN 단계에서 그 수정의 패턴을 분석한다.

```
관찰 예시:
  사용자: "설명이 너무 길어. 코드만 보여줘."

추론:
  - 코드 설명 상세도: detailed → minimal 조정
  - 현재 preferences.md 값 확인
  - 이미 minimal이면: 강화 (pain-points.md에 "긴 설명" 추가)
  - 처음 발생이면: preferences.md 업데이트
```

### 자동 학습: 우회 행동 감지

사용자가 오케스트레이터를 무시하고 직접 에이전트에게 지시하는 패턴:

```
관찰 예시:
  오케스트레이터가 여러 선택지를 제시하는 질문을 했는데
  사용자가 "그냥 네가 알아서 해줘"라고 응답

추론:
  - 질문 선호 형식: choices → open-ended (또는 아예 질문 없이 진행)
```

### 명시적 학습: 직접 피드백

사용자가 명확하게 선호도를 표현할 때:

```
트리거 문구:
  "이렇게 하지 마"       → pain-points.md에 추가
  "이게 좋아"           → preferences.md 강화
  "앞으로도 이렇게 해줘" → preferences.md 업데이트 + 확인 요청
  "내 스타일 기억해줘"   → 현재 세션 관찰 내용 즉시 저장
```

### 확인 요청 프로토콜

새 선호도를 처음 추론했을 때 오케스트레이터가 확인을 구한다:

```
오케스트레이터: "코드 설명 없이 결과만 원하시는 것 같습니다.
앞으로도 이렇게 진행할까요? (예/아니오/이번만)"

예       → preferences.md 업데이트
아니오   → 이번 수정만 적용, 학습하지 않음
이번만   → 이번만 적용, 프로파일 미변경
```

같은 패턴이 2회 이상 반복되면 확인 없이 자동 업데이트한다.

## 적용 메커니즘

### ASK 단계: 질문 형식 조정

질문 선호 형식에 따라 오케스트레이터의 질문 방식이 달라진다:

| 선호 형식 | 질문 방식 |
|----------|----------|
| `choices` | "A 방법과 B 방법 중 어느 것을 선호하십니까?" |
| `open-ended` | "어떤 방식으로 구현하면 좋을까요?" |
| `minimal` | 질문 없이 가장 합리적인 선택 후 진행, 결과만 보고 |

### BUILD 단계: 코딩 스타일 적용

에이전트가 코드를 작성할 때 DNA 프로파일을 참조:

```
주력 언어가 TypeScript인 경우:
  → JavaScript 코드 작성 시 TypeScript로 제안

선호 프레임워크가 React인 경우:
  → 프레임워크 선택이 필요하면 React 우선 추천

코딩 스타일이 "간결"인 경우:
  → 불필요한 추상화 피하기
  → 변수명 짧게 (하지만 명확하게)
  → 인라인 처리 선호

테스트 스타일이 "tdd"인 경우:
  → 구현 전 테스트 코드 먼저 작성
```

### 보고 형식 조정

결과물 보고 시 선호 형식 적용:

| 보고 형식 | 설명 |
|----------|------|
| `bullet` | 짧은 글머리 목록, 핵심만 |
| `prose` | 서술형, 맥락과 배경 포함 |
| `table` | 비교/정리 위주, 표 형태 |

이모지 사용 여부도 적용: 선호하지 않는 경우 출력에서 완전히 제거.

## 진화 트리거

다음 조건 중 하나 충족 시 DNA 프로파일 업데이트:

### 트리거 1: 패턴 반복 (자동)

같은 유형의 수정 요청이 2회 이상 반복되면 확인 없이 자동 학습.

```
예시: 두 세션 연속으로 "너무 자세하게 설명하지 않아도 돼"
→ 코드 설명 상세도: minimal로 업데이트
```

### 트리거 2: 우회 행동 (자동)

사용자가 오케스트레이터를 건너뛰고 에이전트에 직접 지시:

```
예시: 오케스트레이터의 계획 확인 요청을 무시하고 "그냥 만들어줘"
→ 자동화 수준: selective → maximum 방향으로 조정 검토
```

### 트리거 3: 명시적 피드백 (즉시)

사용자의 직접적인 선호 표현 → 즉시 프로파일 업데이트.

## LEARN 단계 실행 절차

세션 종료 시 LEARN 단계에서 pulse-dna는:

```
1. 이번 세션의 상호작용 요약 수집:
   - 수정 요청 목록
   - 거부/수락된 제안
   - 사용자 직접 입력 패턴
   - 진행 중 우회 행동

2. 각 관찰을 프로파일 카테고리에 매핑

3. 기존 프로파일과 비교:
   - 같은 방향: 신뢰도 강화 (recurrence +1)
   - 반대 방향: 충돌 기록, 확인 후 업데이트
   - 신규: 초안으로 추가 (confirmed: false)

4. 업데이트 요약 보고:
   - 업데이트된 선호도 목록
   - 확인이 필요한 새 관찰 목록
```

## 운영 원칙

1. **관찰 기반**: 추정이 아닌 실제 행동에서 추론. "아마 이걸 좋아할 것"은 기록하지 않음.
2. **확인 우선**: 새 선호도는 반드시 한번 확인을 거침 (2회 반복 전까지).
3. **명시성**: 프로파일에 기록된 내용은 사용자가 읽을 수 있는 형태로.
4. **삭제 가능**: 사용자가 언제든 특정 항목 또는 전체 프로파일 초기화 가능.
5. **팀 분리**: 프로파일은 개인 단위. `.gitignore`에 `.pulse/user-dna/` 추가 필수.
6. **세션 분리 전 저장**: 세션을 끊기 전 반드시 LEARN 단계에서 DNA 저장을 완료한다. 세션 분리 직전 저장 여부를 확인하지 않으면 이번 세션 관찰이 소실된다.
