# Profile Schema: 사용자 DNA 프로파일 스키마 상세

---

## 파일 구조 개요

```
.pulse/user-dna/
├── preferences.md       ← 커뮤니케이션 스타일, 설명 상세도, 보고 형식
├── domain-knowledge.md  ← 기술 영역별 이해도
├── pain-points.md       ← 원하지 않는 패턴, 반복 불만
└── decision-history.md  ← 과거 선택, 의사결정 패턴
```

---

## preferences.md 스키마

```markdown
# 사용자 선호도 프로파일

> 마지막 업데이트: {YYYY-MM-DD}
> 관찰 세션 수: {숫자}

## 커뮤니케이션

- 코드 설명 상세도: {minimal|moderate|detailed}
  - minimal: 코드만 제공, 설명 최소화
  - moderate: 핵심 결정 사항만 설명
  - detailed: 배경, 이유, 대안까지 설명
  - 확인됨: {yes|no} / 마지막 관찰: {YYYY-MM-DD}

- 질문 선호 형식: {choices|open-ended|minimal|none}
  - choices: 선택지 제시 ("A vs B 중 선택")
  - open-ended: 열린 질문 ("어떻게 하면 좋을까요?")
  - minimal: 꼭 필요한 것만 질문
  - none: 질문 없이 최선을 선택 후 진행
  - 확인됨: {yes|no} / 마지막 관찰: {YYYY-MM-DD}

- 보고서 형식: {bullet|prose|table|mixed}
  - 확인됨: {yes|no} / 마지막 관찰: {YYYY-MM-DD}

- 이모지 사용: {yes|no}
  - 확인됨: {yes|no}

## 기술 성향

- 주력 언어: [{목록}]
  - 예: [TypeScript, Python]
  - 확인됨: {yes|no}

- 선호 프레임워크: [{목록}]
  - 예: [Next.js, FastAPI, Prisma]
  - 확인됨: {yes|no}

- 테스트 스타일: {tdd|후작성|없음}
  - tdd: 테스트 먼저, 구현 나중
  - 후작성: 구현 후 테스트 추가
  - 없음: 테스트 생략 선호
  - 확인됨: {yes|no}

- 코딩 스타일: {간결|명시적|혼합}
  - 간결: 짧은 변수명, 인라인 처리, 추상화 최소
  - 명시적: 긴 변수명, 타입 명시, 단계별 분리
  - 확인됨: {yes|no}

## 의사결정 패턴

- 속도 vs 품질: {speed|quality|balanced}
  - speed: 빠른 프로토타입, 나중에 리팩토링
  - quality: 처음부터 제대로
  - balanced: 상황에 따라
  - 확인됨: {yes|no}

- 새 기술 vs 검증된 기술: {bleeding-edge|stable|pragmatic}
  - bleeding-edge: 최신 기술 우선 시도
  - stable: 검증된 기술 선호
  - pragmatic: 프로젝트 요구에 따라
  - 확인됨: {yes|no}

- 자동화 수준: {maximum|selective|manual}
  - maximum: 오케스트레이터가 알아서 결정
  - selective: 중요한 것만 확인
  - manual: 매 단계 직접 확인
  - 확인됨: {yes|no}

## 관찰 이력

| 날짜 | 관찰 | 추론 | 확인 여부 |
|------|------|------|-----------|
| {YYYY-MM-DD} | {관찰 내용} | {추론한 선호도} | {yes|no|pending} |
```

## domain-knowledge.md 스키마

```markdown
# 도메인 지식 프로파일

> 마지막 업데이트: {YYYY-MM-DD}

## 잘 아는 영역 (설명 생략 가능)

| 영역 | 이유 | 신뢰도 |
|------|------|--------|
| {예: React 컴포넌트 패턴} | {예: 직접 수정 요청 없음, 고급 개념 사용} | {high|medium|low} |
| {예: SQL 쿼리 최적화} | {예: N+1 문제 직접 발견} | high |

## 모르거나 설명이 필요한 영역

| 영역 | 이유 | 설명 수준 |
|------|------|----------|
| {예: Kubernetes 설정} | {예: 질문이 많았음, 기초 용어 설명 요청} | {basic|intermediate} |
| {예: 함수형 프로그래밍} | {예: 모나드 설명 요청} | basic |

## 주의: 과신 영역

| 영역 | 관찰 | 권고 |
|------|------|------|
| {예: 보안 설정} | {예: SQL 인젝션 방지 코드 오류 발생} | 보안 관련 코드는 검토 제안 |
```

```markdown
## 기술 선택 이력

| 날짜 | 선택 상황 | 선택한 것 | 거부한 것 | 이유 (추정) |
|------|----------|-----------|---------|------------|
| 2024-03-10 | ORM 선택 | Prisma | TypeORM | 타입 안전성 중시 |
| 2024-04-01 | 상태 관리 | Zustand | Redux | 단순함 선호 |
| 2024-04-20 | 스타일링 | Tailwind | CSS Modules | 빠른 개발 선호 |

## 반복 패턴

| 패턴 | 발생 횟수 | 해석 |
|------|---------|------|
| 라이브러리보다 직접 구현 선호 | 3 | 의존성 최소화 성향 |
| 복잡한 설정보다 기본값 선호 | 4 | 단순성 우선 |
| 성능보다 가독성 우선 | 2 | 유지보수성 중시 |

## 되돌린 결정

| 날짜 | 원래 결정 | 변경한 결정 | 이유 |
|------|---------|-----------|------|
| 2024-05-01 | Zustand | Jotai | 아토믹 상태 관리가 더 맞음 |
```
