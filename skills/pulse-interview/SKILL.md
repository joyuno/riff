# Pulse 소크라테스 인터뷰 스킬

소크라테스 인터뷰는 AI가 답을 주는 대신, 해당 도메인 전문가가 던질 법한 질문으로 사용자의 진짜 니즈를 끌어내는 시스템입니다. 프로젝트의 처음(기획)부터 끝(배포/운영)까지 모든 단계에서 올바른 의사결정을 유도합니다.

---

## 언제 이 스킬을 사용하나

다음 키워드나 의도가 감지되면 이 스킬을 활성화합니다:

- "프로젝트 설계해줘", "요구사항 정리", "기획이 필요해"
- "뭘 만들어야 할지 모르겠어", "어떻게 시작해야 해"
- "스펙 정리해줘", "아이디어가 있는데 구체화하고 싶어"
- "MVP 뭐부터 해야 해", "어떤 기술 쓸지 모르겠어"
- Pulse의 ASK 단계 진입 시 자동 호출

---

## 5-Layer 인터뷰 프레임

인터뷰는 5개의 레이어로 구성되며, 각 레이어의 답변이 다음 레이어의 질문을 결정합니다. 상세 질문 트리는 `references/layers.md`를 참조합니다.

### Layer 1: WHY (문제 정의)
> "이 프로젝트가 해결하는 문제가 정확히 뭔가?"

프로젝트의 존재 이유를 명확히 합니다. 이 레이어를 건너뛰면 나머지 모든 결정이 흔들립니다. 문제가 명확하지 않으면 기능 목록이 아무리 길어도 방향이 없습니다.

**핵심 질문:**
- Q1: "이 프로젝트를 만들지 않으면 어떤 일이 벌어지나요?"
- Q2: "이 프로젝트의 성공을 판단할 사람은 누구이고, 그 사람이 '성공'이라고 말하려면 어떤 일이 일어나야 하나요?"

**자동 태깅:**
- 기존 방식이 느리다 → `performance_critical`
- 수동 실수가 많다 → `reliability_critical`
- 새 제품/시장 개척 → `market_risk`

---

### Layer 2: WHO (사용자/이해관계자)
> "누가 쓰고, 누가 돈을 내고, 누가 판단하나?"

사용자를 모르면 기능 우선순위를 잘못 잡습니다. 특히 "쓰는 사람"과 "돈 내는 사람"이 다를 때 두 집단 모두를 분석해야 합니다.

**핵심 질문:**
- Q3: "핵심 사용자를 한 문장으로 묘사해주세요. 직업, 나이, 기술 수준 포함."
- Q4: "그 사용자가 현재 가장 짜증나는 순간은 언제인가요? 구체적인 상황으로."
- Q5: "동시에 서비스를 사용할 최대 사용자 수를 예상하면?"
  - 선택지 A: 1~10명 (팀 내부 도구)
  - 선택지 B: 100~1,000명 (중소 서비스)
  - 선택지 C: 10,000명 이상 (공개 서비스/스케일 필요)

---

### Layer 3: WHAT (범위/우선순위)
> "MVP에 반드시 들어갈 것 vs 나중에 할 것?"

기능 욕심은 프로젝트를 죽입니다. 이 레이어는 "무엇을 하지 않을지" 결정하는 것이 핵심입니다.

**핵심 질문:**
- Q6: "기능 목록을 세 가지로 분류해주세요: 반드시 있어야 함 / 있으면 좋음 / 나중에. 각각 최대 3개씩."
- Q7: "MVP 첫 버전을 언제까지 출시하고 싶으신가요?"
  - 선택지 A: 2주 이내 → 에이전트 수 최대화, 병렬화 극대화
  - 선택지 B: 1~2개월 → 표준 팀 구성
  - 선택지 C: 3개월 이상 → 품질/테스트 강화 전략

---

### Layer 4: HOW (기술 아키텍처)
> "어떤 스택, 어떤 구조, 어떤 트레이드오프?"

기술 결정은 비즈니스 결정입니다. "최신 기술"이 아니라 "팀이 유지할 수 있는 기술"을 선택해야 합니다.

**핵심 질문:**
- Q8: "현재 팀에서 가장 익숙한 기술 스택은 무엇인가요? 없다면 제약 조건은?"
- Q9: [Layer 1 태깅에 따라 동적으로 생성]
  - `performance_critical` 태그 → "응답 시간이 중요한가요, 처리량이 중요한가요? 동시 요청 vs 빠른 단일 응답."
  - `reliability_critical` 태그 → "99.9% vs 99.99% 가용성 중 어느 쪽이 필요한가요? 그 차이가 비용에 미치는 영향을 알고 계신가요?"
  - `market_risk` 태그 → "빠른 피드백(서버리스/관리형)과 장기 통제권(자체 인프라) 중 어느 쪽이 더 중요한가요?"

---

### Layer 5: MEASURE (성공 기준/평가)
> "뭘 측정해야 성공인지 알 수 있나?"

측정할 수 없으면 개선할 수 없습니다. Q2의 답변(성공 판단자)과 연결해서 구체적인 숫자로 만듭니다.

**핵심 질문:**
- Q10: [Q2의 답변 기반] "그 성공을 어떻게 측정할 수 있을까요? 아래 중 어느 방식이 맞나요?"
  - 선택지 A: 사용량 지표 (DAU, 요청 수, 처리 건수)
  - 선택지 B: 비즈니스 지표 (매출, 전환율, 고객 유지율)
  - 선택지 C: 품질 지표 (오류율, 응답시간, 정확도)

---

## 질문 설계 원칙

모든 질문은 다음 기준을 충족해야 합니다:

1. **"예/아니오"가 아닌 "A vs B" 질문**
   - 나쁜 예: "실시간 기능이 필요한가요?"
   - 좋은 예: "1초 지연을 허용하는 준실시간과 100ms 이내 실시간 중 어느 쪽이 실제 사용 시나리오에 맞나요?"

2. **트레이드오프를 노출하는 질문**
   - 모든 기술 결정에는 비용이 있습니다. 질문은 그 비용을 명시해야 합니다.
   - 예: "마이크로서비스는 독립 배포가 가능하지만 운영 복잡도가 3배 증가합니다. 팀 규모가 그 복잡도를 감당할 수 있나요?"

3. **사용자의 과거 경험을 끌어내는 질문**
   - 추상적 답변이 아닌 구체적 경험을 요청합니다.
   - 예: "비슷한 프로젝트를 해본 적 있다면, 그때 가장 후회했던 기술 결정은?"

4. **답변이 아키텍처 결정에 직접 연결되는 질문**
   - 인터뷰의 모든 답변은 `references/answer-to-arch.md`의 매핑 테이블을 통해 구체적인 아키텍처 결정으로 변환됩니다.

5. **각 질문에 "왜 이걸 묻는지" 설명 포함**
   - 사용자가 질문의 의도를 이해할 때 더 정확한 답변을 줍니다.

## 도메인 전문가 모드

인터뷰 시작 시 사용자의 프로젝트 설명에서 도메인을 자동 감지합니다.

### 감지 → 로드 매핑

| 감지 키워드 | 로드할 파일 |
|---|---|
| 웹, React, Next.js, Vue, 웹사이트, 랜딩페이지 | `domains/web-development.md` |
| 스마트스토어, 드롭쉬핑, 쿠팡, 네이버쇼핑, 셀러, 도매 | `domains/smart-store.md` |
| 유튜브, 영상, 릴스, 숏폼, 대본, 편집, 썸네일 | `domains/video-creation.md` |
| 퀀트, 자동매매, 트레이딩, 백테스트, 알고리즘매매 | `domains/quant-trading.md` |
| AI, LLM, GPT, Claude, RAG, 파인튜닝, 에이전트 | `domains/ai-engineering.md` |

### 폴백 전략

도메인이 감지되지 않으면:
1. Layer 4 Q8에서 기술 스택을 물어본 뒤 도메인을 재감지합니다.
2. 그래도 매칭되지 않으면 5-Layer 범용 질문으로 진행합니다.
3. 인터뷰 완료 후 "이 도메인에 특화된 추가 질문이 필요하신가요?" 안내를 제공합니다.

## 인터뷰 진행 방식

### 단계별 흐름

```
1. 도입 (30초)
   → 프로젝트 한 줄 소개 요청
   → 도메인 감지 및 전문가 질문 로드

2. Layer 1-2 (핵심 파악, 5분)
   → Q1~Q5 순서대로 진행
   → 각 답변에서 태그 자동 추출

3. Layer 3 (범위 확정, 3분)
   → Q6~Q7
   → 기능 목록 확정 및 우선순위 정렬

4. Layer 4 (기술 결정, 5분)
   → Q8~Q9 (Q9는 Layer 1 태그 기반으로 동적 생성)
   → 도메인 전문가 질문 추가 (해당하는 경우)

5. Layer 5 (성공 기준, 2분)
   → Q10 (Q2 답변 기반으로 동적 생성)

6. 스코어카드 출력
   → references/scorecard.md 형식으로 인터뷰 품질 평가

7. 아키텍처 매핑 출력
   → references/answer-to-arch.md 매핑 테이블 적용
   → 최종 아키텍처 결정 요약 제시
```

### 진행 규칙

- 한 번에 질문 하나만 합니다. 여러 질문을 한꺼번에 던지지 않습니다.
- 답변이 모호하면 "구체적인 예시를 들어주시겠어요?" 한 번 더 파고듭니다.
- 답변이 충분히 구체적이면 바로 다음 질문으로 넘어갑니다.
- Layer 간 이동 시 "Layer 1이 끝났습니다. 이제 사용자에 대해 이야기해볼까요?" 식으로 전환을 알립니다.
- 총 인터뷰 시간은 15~20분을 목표로 합니다.

## 출력 형식

인터뷰 완료 후 다음 두 가지를 출력합니다:

### 1. 인터뷰 스코어카드
`references/scorecard.md`의 형식을 따릅니다.

### 2. 아키텍처 결정 요약

```markdown
## 인터뷰 결과: 아키텍처 결정 요약

### 감지된 태그
- performance_critical, reliability_critical, ...

### 추천 아키텍처
- 시스템 구조: [모놀리스/마이크로서비스/서버리스]
- 에이전트 팀: [수, 구성, 패턴]
- QA 전략: [Tier 수준]
- 예상 Pulse 수: [N개]

### 핵심 트레이드오프
1. [결정 A] — 이유: [Layer X Q Y 답변 기반]
2. [결정 B] — 이유: [Layer X Q Y 답변 기반]

### 다음 단계
- [ ] 확정된 기능 목록으로 Pulse DNA 생성
- [ ] 아키텍처 결정으로 에이전트 팀 구성
- [ ] 성공 기준으로 QA Tier 설정
```
