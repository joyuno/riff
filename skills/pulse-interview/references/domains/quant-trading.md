### QT-Q1: 전략 유형 및 시장 가설

**질문:**
> "어떤 시장 가설을 기반으로 전략을 만들 건가요? 과거에 오른 종목이 계속 오른다(모멘텀), 많이 떨어진 건 반등한다(평균회귀), 두 자산 간 가격 차이를 이용한다(차익거래), 특정 팩터(저PBR, 저PER 등)에 베팅한다(팩터), 아니면 ML 모델로 패턴을 찾는다(ML 기반) 중 어느 것인가요?"

**왜 이 질문이 중요한가:**
전략 유형이 데이터 요구사항, 실행 빈도, 필요 인프라를 결정합니다. 모멘텀은 일봉 데이터로 충분하지만, 초단타 차익거래는 틱 데이터와 초저지연 인프라가 필요합니다.

**관련 Layer:** WHY, HOW

**답변 분기:**

| 전략 유형 | 데이터 요구 | 실행 빈도 | 인프라 |
|---|---|---|---|
| 모멘텀 | 일봉 OHLCV | 일 1회~주 1회 | 저사양 서버 |
| 평균회귀 | 일봉~분봉 | 일 1회~일 수십 회 | 중간 |
| 차익거래 | 실시간 호가 | 초 단위 | 저지연 서버, 거래소 인접 |
| 팩터 투자 | 재무 데이터 + 가격 | 월~분기 1회 리밸런싱 | 저사양, 배치 처리 |
| ML 기반 | 대용량 피처 | 다양 | GPU 학습 서버 + 추론 서버 |

### QT-Q2: 자산 클래스 및 거래소/브로커

**질문:**
> "어떤 자산을 거래하나요? 한국 주식, 미국 주식, 암호화폐, 선물/옵션 중 어느 것인가요? 그리고 이미 사용 중인 증권사나 거래소가 있나요? API 신청을 완료했나요?"

**왜 이 질문이 중요한가:**
자산 클래스와 거래소가 사용 가능한 API를 결정합니다. 한국 주식은 키움/이베스트 API가 Windows 전용이라는 중요한 제약이 있습니다.

**관련 Layer:** HOW

**주요 API 현황:**

| 거래소/브로커 | API 품질 | OS 제약 | 주의사항 |
|---|---|---|---|
| 키움증권 (OpenAPI+) | 중간 | Windows 전용 | Wine 또는 VM 필요 (Linux/Mac 사용 시) |
| 이베스트투자증권 (xingAPI) | 중간 | Windows 전용 | REST 버전 일부 제공 |
| 한국투자증권 (KIS API) | 높음 | 크로스플랫폼 | REST API, 상대적으로 현대적 |
| Binance | 매우 높음 | 크로스플랫폼 | WebSocket 실시간 지원, 풍부한 문서 |
| Bybit | 높음 | 크로스플랫폼 | 선물 특화, 한국어 지원 |
| Interactive Brokers (IBKR) | 높음 | 크로스플랫폼 | 미국/글로벌 주식, TWS API |
| Alpaca | 높음 | 크로스플랫폼 | 미국 주식, Paper Trading 지원 |

**키움 API 제약 경고:**
> 키움 OpenAPI+는 Windows COM 객체 기반입니다. Linux 서버에서 실행하려면 Wine, Docker+Wine, 또는 Windows VM이 필요합니다. 이 제약을 알고 계신가요? 한국투자증권 REST API로 대체 가능한지 검토를 권장합니다.

### QT-Q3: 데이터 소스 및 품질

**질문:**
> "백테스트와 실전매매에 사용할 데이터 소스는 어디인가요? 무료 데이터(야후파이낸스, FinanceDataReader)를 쓸 건가요, 유료 데이터 구독 서비스를 사용할 건가요? 대안 데이터(뉴스 감성, 소셜미디어, 공시 텍스트)도 활용하나요?"

**왜 이 질문이 중요한가:**
데이터 품질이 전략 성과를 결정합니다. 무료 데이터는 생존 편향(Survivorship Bias), 포인트-인-타임(Point-in-Time) 오류, 누락 데이터 문제가 있어 백테스트 결과가 과대평가될 수 있습니다.

**관련 Layer:** HOW

**데이터 소스 비교:**

| 소스 | 비용 | 품질 | 사용 가능 자산 | 주의사항 |
|---|---|---|---|---|
| FinanceDataReader | 무료 | 중간 | 한국/미국 주식 | 생존 편향 주의 |
| Yahoo Finance (yfinance) | 무료 | 중간 | 글로벌 | 수시로 API 변경 |
| KRX 공식 데이터 | 무료 | 높음 | 한국 주식 | 일별, 실시간 아님 |
| Binance 공식 API | 무료 | 높음 | 암호화폐 | 과거 데이터 제한 |
| Quantiacs / Quandl | 유료 | 높음 | 글로벌 | 팩터 데이터 포함 |
| Bloomberg / Refinitiv | 매우 유료 | 매우 높음 | 전 자산 | 기관급 |

### QT-Q4: 백테스트 프레임워크

**질문:**
> "백테스트를 어떤 방식으로 구현할 건가요? 기존 프레임워크(Backtrader, Zipline, QuantConnect, VectorBT)를 사용할 건가요, 직접 구현할 건가요? 워크포워드(Walk-forward) 테스트나 몬테카를로 시뮬레이션도 필요한가요?"

**왜 이 질문이 중요한가:**
백테스트 방법론이 전략의 신뢰도를 결정합니다. 단순 백테스트는 과적합(Overfitting) 위험이 있습니다. 올바른 프레임워크 선택과 검증 방법론이 없으면 실전에서 전략이 무너집니다.

**관련 Layer:** HOW

**프레임워크 비교:**

| 프레임워크 | 장점 | 단점 | 적합한 전략 |
|---|---|---|---|
| Backtrader | 직관적, 한국어 자료 많음 | 느림, 오래된 코드베이스 | 일봉~분봉 전략 |
| VectorBT | 매우 빠름 (벡터화) | 학습 곡선 높음 | 대용량 파라미터 최적화 |
| Zipline (Ralphinex fork) | 실전매매 연동 | 유지보수 중단 | 주식 팩터 전략 |
| QuantConnect (Lean) | 완성도 높음, 클라우드 | 로컬 설정 복잡 | 멀티 자산, 복잡한 전략 |
| 직접 구현 (pandas) | 완전한 통제 | 시간 많이 걸림 | 단순 전략 프로토타이핑 |

**과적합 방지 경고:**
> 단일 기간 백테스트로 나온 좋은 결과는 신뢰하기 어렵습니다. 워크포워드 검증과 아웃오브샘플(Out-of-Sample) 테스트를 반드시 포함해야 합니다. 이 부분을 시스템에 포함할 건가요?

### QT-Q5: 실행 주기 및 지연 민감도

**질문:**
> "전략이 얼마나 자주 매매 신호를 생성하고 실행해야 하나요? 하루 한 번 리밸런싱인가요, 분 단위 매매인가요, 초 단위 또는 더 빠른 고빈도 매매인가요?"

**왜 이 질문이 중요한가:**
실행 주기가 인프라 비용과 슬리피지/수수료 영향을 결정합니다. 고빈도 매매는 수수료가 수익을 잠식할 수 있고, 코로케이션 서버가 필요할 수 있습니다.

**관련 Layer:** HOW

**답변 분기:**

| 실행 주기 | 인프라 요구 | 수수료 영향 | 슬리피지 |
|---|---|---|---|
| 일 1회 (종가 매매) | 저사양 (크론잡) | 낮음 | 무시 가능 |
| 분봉 (1-60분) | 중간 (상시 서버) | 중간 | 중간 |
| 초봉 (1-60초) | 높음 (저지연 서버) | 높음 | 중요한 변수 |
| 틱 단위 (고빈도) | 매우 높음 (코로케이션) | 매우 높음 | 핵심 변수 |

**수수료 시뮬레이션 필수 여부:**
분봉 이상 빈도에서는 반드시 수수료와 슬리피지를 백테스트에 포함해야 합니다. 포함하지 않으면 백테스트 수익률이 실전과 크게 다릅니다.

### QT-Q6: 리스크 관리 및 포지션 사이징

**질문:**
> "손실을 얼마나 허용할 수 있나요? 전략 전체의 최대 낙폭(Max Drawdown) 한도와 단일 포지션 최대 손실(Stop-Loss)을 정해두셨나요? 포지션 크기는 어떻게 결정할 건가요?"

**왜 이 질문이 중요한가:**
리스크 관리가 없는 자동매매 시스템은 언제든 계좌를 날릴 수 있습니다. 이것이 전략 자체보다 더 중요한 경우가 많습니다.

**관련 Layer:** HOW, MEASURE

**리스크 관리 요소:**

| 요소 | 설명 | 구현 |
|---|---|---|
| 최대 낙폭 한도 | 전략 전체 손실 한도 도달 시 자동 중지 | 모니터링 + 자동 청산 |
| 포지션 스탑로스 | 단일 포지션 손실 한도 | 조건부 주문 또는 모니터링 루프 |
| 포지션 사이징 | Kelly Criterion, Fixed Fraction, Equal Weight | 주문 수량 계산 모듈 |
| 상관관계 리스크 | 동일 방향 포지션 과다 집중 방지 | 포트폴리오 상관계수 모니터링 |
| 레버리지 한도 | 선물/마진 사용 시 레버리지 캡 | 주문 전 검증 레이어 |

### QT-Q7: 실시간 모니터링 및 알림

**질문:**
> "전략이 돌아가는 동안 어떤 것들을 모니터링해야 하나요? 포지션 현황, 수익/손실, 오류 발생, 비정상적인 시장 상황(서킷브레이커, 거래 정지) 중 어느 것이 가장 중요하고, 알림은 어떻게 받고 싶으신가요?"

**왜 이 질문이 중요한가:**
자동매매 시스템은 24시간 돌아가지만 개발자는 항상 볼 수 없습니다. 이상 감지와 알림 시스템이 없으면 오류가 발생해도 며칠 후에 알게 됩니다.

**관련 Layer:** MEASURE

**모니터링 요소별 구현:**

| 모니터링 항목 | 알림 방식 | 구현 |
|---|---|---|
| 포지션 현황 | 텔레그램 봇 (주기적 리포트) | 텔레그램 Bot API |
| 오류/예외 | 즉시 알림 (텔레그램/슬랙) | try/except + 알림 |
| P&L 요약 | 일일 리포트 | 크론잡 + 메시지 |
| 비정상 시장 | 즉시 중단 + 알림 | 거래량/변동성 모니터링 |
| API 연결 끊김 | 즉시 알림 | 헬스체크 루프 |

### QT-Q8: 규제 및 세금 고려사항

**질문:**
> "자동매매와 관련한 법적/세금 이슈를 검토하셨나요? 한국 주식 양도소득세, 암호화폐 과세, 금융투자업 등록 여부, 해외 선물 신고 등을 고려해야 하는 상황인가요?"

**왜 이 질문이 중요한가:**
수익이 나도 세금 처리가 잘못되면 문제가 생깁니다. 특히 법인이 자동매매를 하는 경우 금융투자업 등록 요건을 확인해야 합니다.

**관련 Layer:** WHY

**주요 규제/세금 이슈:**

| 항목 | 현황 (2024년 기준) | 주의사항 |
|---|---|---|
| 국내 주식 양도소득 | 대주주(10억+)만 과세 | 소액은 비과세 (2025년 금투세 유예) |
| 암호화폐 소득세 | 2025년부터 250만원 초과분 22% | 거래 내역 보관 필수 |
| 해외 주식 양도소득 | 연 250만원 초과분 22% | 5월 종합소득세 신고 |
| 해외 선물 | 소득세 신고 | 환차익도 포함 |
| 법인 자동매매 | 금융투자업 요건 검토 필요 | 타인 자금 운용 시 인가 필요 |

**면책 고지:** 세금/법률 사항은 전문가(세무사/변호사) 확인을 권장합니다. 이 정보는 참고용입니다.
