### WD-Q1: 렌더링 전략 — SSR vs CSR vs SSG

**질문:**
> "이 서비스의 페이지들을 생각해봤을 때, SEO가 중요한 공개 페이지(상품 목록, 블로그, 랜딩)와 로그인 후 개인화 페이지(대시보드, 마이페이지) 중 어느 쪽이 더 많은가요?"

**왜 이 질문이 중요한가:**
렌더링 전략이 프레임워크 선택과 인프라 비용을 결정합니다. 잘못 선택하면 SEO가 망하거나 서버 비용이 폭발합니다.

**관련 Layer:** HOW

**답변 분기:**

| 답변 | 추천 전략 | 아키텍처 영향 |
|---|---|---|
| SEO 공개 페이지 중심 | SSR 또는 SSG | Next.js 추천, Vercel 배포, ISR 적용 |
| 개인화 대시보드 중심 | CSR (SPA) | React + Vite, 정적 호스팅 가능, API Gateway 패턴 |
| 둘 다 비슷하게 있음 | 하이브리드 (Next.js App Router) | 페이지별 전략 혼용, 복잡도 증가 인지 필요 |
| 랜딩 + 마케팅 사이트만 | SSG | Astro 또는 Next.js 정적 내보내기, CDN 배포 |

### WD-Q2: 인증 방식 선택

**질문:**
> "사용자가 어떻게 로그인하나요? 이메일/비밀번호만인가요, 구글/카카오/네이버 소셜 로그인도 필요한가요? 그리고 기업 고객이라면 사내 SSO(Single Sign-On)도 고려해야 하나요?"

**왜 이 질문이 중요한가:**
인증은 나중에 바꾸기 가장 어려운 결정 중 하나입니다. 소셜 로그인 추가는 생각보다 복잡하고, SSO는 완전히 다른 구현입니다.

**관련 Layer:** HOW

**답변 분기:**

| 답변 | 추천 솔루션 | 구현 비용 |
|---|---|---|
| 이메일/비밀번호만 | 자체 구현 또는 NextAuth.js | 낮음 |
| 소셜 로그인 필요 | NextAuth.js / Supabase Auth / Clerk | 중간 |
| 소셜 + SSO + 기업 | Auth0 또는 Clerk | 높음 (월 비용 발생) |
| 간단하고 빨리 | Supabase Auth (이메일+소셜 통합) | 낮음, 관리형 |

### WD-Q3: 데이터베이스 및 백엔드 구조

**질문:**
> "데이터 관계가 복잡한가요? 예를 들어 '사용자-주문-상품-리뷰'처럼 여러 테이블이 JOIN되는 구조인가요, 아니면 설문 응답이나 로그처럼 단순한 문서/레코드 저장인가요?"

**왜 이 질문이 중요한가:**
데이터 구조가 DB 선택을 결정합니다. 관계형 데이터를 NoSQL에 억지로 넣으면 쿼리가 지옥이 됩니다.

**관련 Layer:** HOW

**답변 분기:**

| 데이터 특성 | 추천 DB | 백엔드 패턴 |
|---|---|---|
| 복잡한 관계형 | PostgreSQL (Supabase 또는 직접) | ORM 필수 (Prisma / Drizzle) |
| 단순 문서 저장 | Firestore 또는 MongoDB | 서버리스 친화적 |
| 빠른 출시 + 관계형 | Supabase | BaaS 활용, 백엔드 최소화 |
| 대용량 비정형 | MongoDB Atlas | 유연한 스키마 필요 시 |

### WD-Q4: 실시간 기능 요구사항

**질문:**
> "사용자가 페이지를 새로고침하지 않아도 데이터가 업데이트되어야 하는 기능이 있나요? 예를 들어 채팅, 알림, 실시간 대시보드, 협업 편집 같은 것들이요."

**왜 이 질문이 중요한가:**
실시간 기능은 아키텍처를 완전히 바꿉니다. WebSocket 서버는 서버리스와 호환되지 않고, 인프라 비용이 다릅니다.

**관련 Layer:** WHAT, HOW

**답변 분기:**

| 실시간 필요 여부 | 방식 | 인프라 영향 |
|---|---|---|
| 없음 (폴링으로 충분) | HTTP 폴링 (30초~1분 간격) | 기존 인프라 그대로 |
| 약한 실시간 (수초 지연 허용) | Server-Sent Events (SSE) | 서버리스 가능, 단방향 |
| 강한 실시간 (채팅/협업) | WebSocket | 항상 켜진 서버 필요, 서버리스 불가 |
| Supabase 사용 시 | Supabase Realtime | 내장 기능 활용 가능 |

### WD-Q5: 상태 관리 전략

**질문:**
> "클라이언트 측에서 얼마나 많은 상태를 관리해야 하나요? 단순히 서버 데이터를 표시하는 수준인가요, 아니면 복잡한 사용자 인터랙션(멀티스텝 폼, 실시간 편집, 오프라인 모드)이 필요한가요?"

**왜 이 질문이 중요한가:**
상태 관리 라이브러리 선택이 개발 복잡도와 성능에 큰 영향을 미칩니다. 단순한 서비스에 Redux를 쓰면 오버엔지니어링입니다.

**관련 Layer:** HOW

**답변 분기:**

| 복잡도 | 추천 솔루션 | 비고 |
|---|---|---|
| 단순 (서버 데이터 표시) | React Query / SWR | 서버 상태만 관리, 클라이언트 상태 최소화 |
| 중간 (폼 + 일부 복잡 UI) | Zustand + React Query | 경량, 학습 곡선 낮음 |
| 복잡 (대규모 SPA) | Redux Toolkit + RTK Query | 예측 가능성 높음, 보일러플레이트 많음 |
| Next.js + 서버 컴포넌트 | 서버 상태는 서버 컴포넌트로 | 클라이언트 상태 최소화 가능 |

### WD-Q6: SEO 요구사항

**질문:**
> "구글 검색 결과에 노출되어야 하는 페이지가 있나요? 있다면 어떤 페이지들이고, 검색 트래픽이 사업에 얼마나 중요한가요?"

**왜 이 질문이 중요한가:**
SEO가 사업의 핵심이라면 CSR(React SPA)은 처음부터 선택지에서 제외해야 합니다. 나중에 SSR로 전환하는 비용은 매우 큽니다.

**관련 Layer:** WHY, HOW

**답변 분기:**

| SEO 중요도 | 결정 |
|---|---|
| 핵심 (검색 트래픽이 주 유입) | SSR 또는 SSG 필수, Next.js 강력 추천 |
| 중간 (일부 페이지만 필요) | 하이브리드: 공개 페이지 SSR + 대시보드 CSR |
| 없음 (로그인 후 서비스) | CSR 가능, SPA로 충분 |

### WD-Q7: 배포 환경 및 DevOps

**질문:**
> "이 서비스를 어디에 배포할 계획인가요? 특정 클라우드 벤더를 써야 하는 제약이 있나요? 그리고 배포를 자동화(CI/CD)할 계획인가요, 아니면 수동 배포로 시작할 건가요?"

**왜 이 질문이 중요한가:**
배포 환경이 프레임워크 선택과 운영 비용을 결정합니다. Vercel은 Next.js에 최적화되어 있지만 비용 구조가 다릅니다.

**관련 Layer:** HOW

**답변 분기:**

| 상황 | 추천 환경 | 월 비용 기준 |
|---|---|---|
| 제약 없음 + Next.js | Vercel | 무료 ~ $20+ |
| 제약 없음 + 다양한 스택 | Railway 또는 Render | $5 ~ $50 |
| AWS 사용 필요 | ECS + RDS + CloudFront | $50 ~ |
| 자체 서버 보유 | Docker + Nginx | 서버 비용만 |
| 빠른 시작 + 정적 | Netlify 또는 Cloudflare Pages | 무료 ~ |
