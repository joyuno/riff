# 5-Layer 인터뷰 질문 트리

각 Layer의 답변이 다음 질문을 결정하는 분기형 질문 트리입니다. 모든 질문은 구체적인 아키텍처 결정으로 연결됩니다.

---

## Layer 1: WHY — 문제 정의

> 목적: 프로젝트가 존재해야 하는 이유를 명확히 한다. 이 레이어를 건너뛰면 나머지 모든 결정의 근거가 사라진다.

---

### Q1: "이 프로젝트를 만들지 않으면 어떤 일이 벌어지나요?"

**왜 이 질문인가:** "무엇을 만드냐"가 아니라 "만들지 않으면 무엇이 잘못되냐"를 물어야 진짜 문제가 드러납니다. 기능 목록이 아닌 고통에서 시작해야 합니다.

**답변 분기 및 자동 태깅:**

| 답변 패턴 | 추출 태그 | 아키텍처 영향 |
|---|---|---|
| "지금 방식이 너무 느려요", "수작업이 시간이 너무 걸려요" | `performance_critical` | 처리 속도 최우선, 비동기 처리 필수, 캐싱 레이어 고려 |
| "실수가 많이 나요", "사람이 하다 보니 오류가 생겨요" | `reliability_critical` | 검증 레이어 필수, 감사 로그 필수, 롤백 메커니즘 설계 |
| "새 시장/제품이에요", "지금은 아무도 안 하는 걸 만들어요" | `market_risk` | 빠른 출시 우선, 피벗 가능한 구조, 유연한 스키마 |
| "경쟁사가 이미 하고 있어요" | `competitive_pressure` | 차별화 기능 집중, 빠른 배포 주기 |
| "규정/법률 때문에 꼭 필요해요" | `compliance_required` | 감사 추적 필수, 데이터 보존 정책 필수, 접근 제어 강화 |

**후속 탐색 질문 (답변이 모호할 때):**
- "그 문제가 매일 발생하나요, 가끔 발생하나요?"
- "그 문제로 인해 지금 어떤 비용(시간/돈/기회)이 발생하고 있나요?"

---

### Q2: "이 프로젝트의 성공을 판단할 사람은 누구이고, 그 사람이 '성공'이라고 말하려면 어떤 일이 일어나야 하나요?"

**왜 이 질문인가:** 성공의 판단자를 명확히 해야 측정 지표를 올바르게 설정할 수 있습니다. "팀장이 좋아하면 성공"과 "DAU 1만 명이면 성공"은 완전히 다른 프로젝트입니다. Q2의 답변은 Layer 5(MEASURE)의 Q10에서 재활용됩니다.

**답변 분기:**

| 성공 판단자 | 성공 기준 유형 | Layer 5 연결 |
|---|---|---|
| 내부 팀원/경영진 | 운영 효율 지표 (시간 절감, 오류 감소) | Q10 → 선택지 A (사용량/처리 건수) |
| 외부 고객/사용자 | 사용자 만족도, 재방문율 | Q10 → 선택지 B (비즈니스 지표) |
| 투자자/이사회 | 매출, 성장률, 단위 경제성 | Q10 → 선택지 B (비즈니스 지표) |
| 기술팀 | 안정성, 배포 빈도, 오류율 | Q10 → 선택지 C (품질 지표) |

**자동 메모:** Q2의 답변을 기록해두고 Layer 5에서 "앞서 말씀하신 [판단자]가 성공이라고 하려면..." 으로 연결합니다.

---

## Layer 2: WHO — 사용자/이해관계자

> 목적: 누구를 위해 만드는지 명확히 한다. 사용자를 모르면 기능 우선순위를 잘못 잡는다.

---

### Q3: "핵심 사용자를 한 문장으로 묘사해주세요. 직업, 나이대, 기술 수준을 포함해서."

**왜 이 질문인가:** "일반 사용자"는 존재하지 않습니다. "40대 소상공인이 스마트폰으로 사용"과 "20대 개발자가 CLI로 사용"은 UI/UX, 오류 처리, 온보딩 방식이 완전히 달라집니다.

**답변 분기:**

| 사용자 프로필 | 영향받는 결정 |
|---|---|
| 비기술자, 고령층 | 복잡한 UI 금지, 오류 메시지 한국어 친절하게, 자동 저장 필수 |
| 일반 직장인 | 웹 중심, 모바일 반응형 필수, 학습 곡선 최소화 |
| 개발자/기술자 | API 우선, CLI 지원, 기술적 오류 메시지 허용 |
| 관리자/의사결정자 | 대시보드 중심, 요약 뷰 필수, 엑스포트 기능 |

---

### Q4: "그 사용자가 현재 가장 짜증나는 순간은 언제인가요? 구체적인 상황으로 설명해주세요."

**왜 이 질문인가:** Q3에서 사용자를 정의했으면 그 사용자의 고통을 구체화해야 합니다. "불편하다"가 아니라 "어떤 상황에서 어떻게 불편한지"를 파악해야 해결책이 나옵니다.

**예시 답변과 영향:**

```
답변: "엑셀 파일을 10개 열어서 데이터를 복붙하다가 실수가 생겨요"
→ 데이터 통합 기능이 핵심, 검증 UI 필수, 배치 처리 지원

답변: "담당자한테 매번 물어봐야 해서 기다리는 시간이 너무 길어요"
→ 셀프서비스 기능 핵심, 지식베이스/FAQ 자동화, 알림 시스템

답변: "아침마다 수동으로 리포트 만들어서 팀장한테 보내요"
→ 자동화/스케줄링이 핵심 기능, 이메일/메신저 연동 필수
```

---

### Q5: "동시에 서비스를 사용할 최대 사용자 수를 예상하면 어느 정도인가요?"

**왜 이 질문인가:** 아키텍처의 기반이 되는 질문입니다. 10명용 시스템과 10,000명용 시스템은 설계가 완전히 다릅니다. 초기에 이 결정을 틀리면 나중에 전면 재설계가 필요합니다.

**선택지 및 아키텍처 영향:**

| 선택지 | 규모 | 추천 아키텍처 | 인프라 |
|---|---|---|---|
| A: 1~10명 | 팀 내부 도구 | 모놀리식, 단일 서버 | 단일 EC2 또는 Railway, SQLite 가능 |
| B: 100~1,000명 | 중소 서비스 | 모놀리식 + CDN | RDS + 캐싱, Vercel/Render |
| C: 10,000명 이상 | 공개 서비스 | 수평 확장 가능 구조 | 로드밸런서, DB 분리, 캐싱 필수 |

**후속 질문 (C 선택 시):**
- "그 사용자 중 동시에 쓰기 작업을 하는 비율은 얼마나 될까요?" (읽기 vs 쓰기 비율이 DB 선택에 영향)

---

## Layer 3: WHAT — 범위/우선순위

> 목적: 무엇을 만들고 무엇을 만들지 않을지 결정한다. 기능 욕심이 프로젝트를 죽인다.

---

### Q6: "기능 목록을 세 가지로 분류해주세요: '반드시 있어야 함' / '있으면 좋음' / '나중에'. 각각 최대 3개씩 말씀해주세요."

**왜 이 질문인가:** "다 필요해요"는 유효한 답변이 아닙니다. 강제로 우선순위를 정하게 해야 진짜 MVP가 드러납니다. 각 3개 제한은 집중을 강제합니다.

**분류 기준 안내 (사용자에게 제공):**
- 반드시 있어야 함: 이게 없으면 서비스 자체가 성립하지 않음
- 있으면 좋음: 경쟁 우위를 줄 수 있지만 없어도 작동함
- 나중에: 좋지만 지금은 아님, 2차 출시에서 고려

**아키텍처 연결:**
- "반드시" 목록 → 데이터 모델의 핵심 엔티티 결정
- "있으면 좋음" 목록 → 확장 가능한 플러그인/모듈 설계 여부 결정
- "나중에" 목록 → 지금 설계에서 미리 고려할 인터페이스 정의

**일관성 검사:**
- Layer 1 태그가 `performance_critical`인데 "있으면 좋음"에 캐싱이 있으면 → 모순 지적
- Layer 2 Q5가 C(10,000명)인데 "반드시"에 실시간 채팅이 있으면 → 복잡도 경고

---

### Q7: "MVP 첫 버전을 언제까지 출시하고 싶으신가요?"

**왜 이 질문인가:** 데드라인이 기술 선택을 결정합니다. 2주 안에 나와야 한다면 새로운 기술 학습은 불가능합니다. 에이전트 수와 병렬화 수준도 이 답변으로 결정됩니다.

**선택지 및 Pulse 전략:**

| 타임라인 | Pulse 전략 | 에이전트 구성 |
|---|---|---|
| A: 2주 이내 | 속도 최우선: 검증된 스택, 최소 기능 | 에이전트 최대화, 모든 작업 병렬화, QA Tier 1 |
| B: 1~2개월 | 균형: 표준 팀 구성 | 표준 에이전트 팀, 핵심 테스트 포함, QA Tier 2 |
| C: 3개월 이상 | 품질 우선: 테스트 강화, 문서화 | 전문화된 에이전트, 풀 QA, 문서 에이전트 포함, QA Tier 3 |

---

## Layer 4: HOW — 기술 아키텍처

> 목적: 어떤 기술로 어떻게 만들지 결정한다. "최신"이 아닌 "적합한" 기술을 선택한다.

---

### Q8: "현재 팀에서 가장 익숙한 기술 스택은 무엇인가요? 없다면 제약 조건(예산, 클라우드 벤더, 언어)은?"

**왜 이 질문인가:** 팀이 유지할 수 없는 기술은 기술 부채입니다. 아무리 좋은 아키텍처도 운영할 사람이 없으면 소용없습니다.

**답변 처리:**

```
기존 스택 있음 → 해당 스택 기준으로 Q9 질문 생성
  예: "Python/Django 팀이에요" → Django REST vs FastAPI vs GraphQL 질문

기존 스택 없음 (신규 팀) → 도메인 + Q5 답변 기반으로 스택 추천
  예: 웹 + B규모 → Next.js + PostgreSQL + Vercel 추천 후 동의 여부 확인

제약 조건 있음 (예: AWS만 가능) → 해당 벤더 서비스 기준으로 아키텍처 제안
```

---

### Q9: [Layer 1 태깅 기반 동적 생성]

**왜 이 질문인가:** 일반적인 기술 질문이 아닌 이 프로젝트의 핵심 트레이드오프를 노출하는 질문을 합니다.

**태그별 질문:**

**`performance_critical` 태그 시:**
> "응답 속도(단일 요청의 빠름)와 처리량(동시 요청 많이) 중 어느 것이 더 중요한가요?
> 응답 속도 우선 → 인메모리 캐싱, 싱글 서버 최적화
> 처리량 우선 → 수평 확장, 큐 기반 비동기 처리"

**`reliability_critical` 태그 시:**
> "99.9% 가용성(월 8.7시간 다운 허용)과 99.99%(월 52분 허용) 중 어느 쪽이 필요한가요?
> 그 차이는 인프라 비용이 약 5~10배 차이납니다."

**`market_risk` 태그 시:**
> "빠른 시장 피드백(관리형 서비스, 빠른 출시)과 장기 기술 통제권(자체 인프라, 느리지만 유연) 중 어느 쪽이 지금 단계에서 더 중요한가요?"

**`compliance_required` 태그 시:**
> "어떤 규정 준수가 필요한가요? GDPR/개인정보보호법/금융규정에 따라 데이터 보존, 암호화, 감사 로그 요구사항이 달라집니다."

**`competitive_pressure` 태그 시:**
> "경쟁사 대비 어떤 한 가지에서 명확히 더 나아야 하나요? 속도/가격/UX/특정 기능 중 선택해주세요. 모든 것에서 이기려 하면 아무것도 이길 수 없습니다."

---

## Layer 5: MEASURE — 성공 기준/평가

> 목적: 측정할 수 없으면 개선할 수 없다. Q2의 판단자와 연결해서 구체적인 숫자로 만든다.

---

### Q10: [Q2 답변 기반 동적 생성]

**왜 이 질문인가:** "성공"을 정의하지 않으면 언제 완성인지 알 수 없습니다. Q2에서 말한 판단자가 보게 될 숫자를 미리 정해야 합니다.

**Q2 답변별 Q10 변형:**

**Q2 판단자가 "내부 팀/경영진"일 때:**
> "[Q4에서 파악한 고통 지점]이 어느 정도 해결되면 성공이라고 볼 수 있을까요?
> A: 처리 시간이 X% 감소 (수치로)
> B: 오류 건수가 월 N건 미만으로 감소
> C: 담당자 작업 시간이 주 N시간 절감"

**Q2 판단자가 "외부 고객"일 때:**
> "고객이 이 서비스를 계속 쓴다는 것을 어떻게 알 수 있을까요?
> A: 월 재방문율 N% 이상
> B: 결제 전환율 N% 이상
> C: NPS(순추천지수) N점 이상"

**Q2 판단자가 "투자자/이사회"일 때:**
> "6개월 후 투자자 보고에서 어떤 숫자를 보여줄 수 있어야 하나요?
> A: MAU N만 명 달성
> B: MRR N만원 달성
> C: 고객 획득 비용(CAC) N원 이하"

**수치 없는 답변 처리:**
답변에 구체적인 숫자가 없으면 반드시 한 번 더 파고듭니다:
> "조금 더 구체적으로 해볼까요? 지금 [현재 상태]가 [목표 상태]로 바뀌면 성공이라고 할 수 있을까요? 그 수치를 함께 정해봐요."

---

## 분기 결정 로직 요약

```
Q1 답변 → 태그 추출 → Q9 내용 결정
Q2 답변 → 판단자 기록 → Q10 내용 결정
Q3 답변 → 사용자 프로필 → UI/UX 제약 결정
Q5 답변 → 규모 선택 → 인프라 아키텍처 결정
Q7 답변 → 타임라인 → Pulse 에이전트 수/병렬화 결정
Q8 답변 → 스택 확인 → 도메인 전문가 질문 방향 결정
```

모든 태그와 답변은 `references/answer-to-arch.md`의 매핑 테이블에 입력되어 최종 아키텍처 결정을 생성합니다.
