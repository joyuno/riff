# Pulse — AI-Native Project Loop

## 핵심 원칙

- 한 번에 잘 만들지 않는다. 빠르게 많이 시도한다.
- 매 Pulse마다 작동하는 결과물이 존재한다.
- 질문은 트레이드오프가 있을 때만 한다.
- 실패는 비용이 아니라 학습이다.
- AI 시간으로 사고한다 (인간 기준 일정을 적용하지 않음).

---

## Pulse 0: 프로젝트 부팅

첫 Pulse는 특별하다. 방향을 잡는 단계이며, 이후 모든 Pulse의 기준이 된다.

**실행 순서:**

1. `pulse-interview` 모듈이 설치되어 있으면 5-Layer 인터뷰를 실행한다.
2. 모듈이 없으면 핵심 질문 2개만 한다:
   - "이 프로젝트가 해결하는 문제는 무엇인가?"
   - "성공하면 어떤 모습인가?"
3. 답변을 바탕으로 다음을 산출한다:
   - 문제 정의 (1-2문장)
   - 핵심 사용자 페르소나
   - 유저 저니 3개 (주요 시나리오)
   - 성공 기준 (측정 가능한 형태)
   - 질문 예산 설정 (A~D 중 사용자 선택)

4. `_workspace/` 디렉토리를 생성하고 `pulse-log.md`를 초기화한다.

**Pulse 0 산출물 구조:**
```
_workspace/
  pulse-log.md          # 전체 Pulse 이력
  pulse-0/
    problem.md          # 문제 정의
    personas.md         # 핵심 사용자
    journeys.md         # 유저 저니 3개
    success-criteria.md # 성공 기준
    question-budget.md  # 질문 예산 설정
```

## Pulse 사이클 (매 반복)

각 Pulse는 다섯 단계를 순서대로 실행한다: ASK → EXPLORE → BUILD → VERIFY → LEARN

---

### 1. ASK (질문)

**목적:** 진행에 필요한 결정을 최소한으로 확인한다.

**실행 규칙:**
- `pulse-interview` 스킬이 있으면 해당 모듈을 호출한다.
- 없으면 현재 단계에서 **트레이드오프가 있는 결정**만 질문한다.
- 확신도가 질문 예산 임계값 이상이면 이 단계를 건너뛴다.

**질문 예산 시스템:**

| 레벨 | 이름 | 설명 | 질문 임계값 |
|------|------|------|------------|
| A | 매 단계 확인 | 모든 결정 전에 승인 요청 | 확신 < 99% |
| B | 핵심만 | 중요 트레이드오프만 질문 | 확신 < 80% |
| C | 최소한 | 방향 전환이 필요할 때만 | 확신 < 60% |
| D | 전자동 | 질문 없이 진행, 완료 후 보고 | 질문 안 함 |

프로젝트 시작 시 사용자에게 레벨을 선택하게 한다. 진행 중에도 변경 가능하다.

---

### 2. EXPLORE (탐색)

**목적:** 구현 방법이 불명확할 때 동시에 여러 방향을 탐색한다.

**실행 규칙:**
- 방법이 2개 이상 존재하고 트레이드오프가 명확하지 않으면 분신술 패턴을 적용한다.
- 방법이 이미 명확하면 이 단계를 건너뛴다.

**분신술 패턴:**
- 각 방향을 별도 에이전트로 스폰하여 동시에 프로토타이핑한다.
- 예: 방향 A 에이전트, 방향 B 에이전트를 병렬 실행 후 결과 비교.

**대립 토론 패턴:**
- 찬성 에이전트: 해당 방향의 장점을 최대한 옹호한다.
- 반대 에이전트: 해당 방향의 약점과 리스크를 파고든다.
- 종합 에이전트: 두 관점을 받아 최종 결정을 내린다.

**미래 시뮬레이션:**
- "이 결정이 6개월 후에도 유효한가?"를 검증한다.
- 확장성, 유지보수성, 팀 온보딩 비용을 사전 점검한다.

**점진적 확신:**
- 탐색 후 확신도가 여전히 낮으면 추가 탐색 사이클을 자동으로 돌린다.
- 최대 3회 탐색 후에도 결정이 안 되면 사용자에게 판단을 요청한다.

---

### 3. BUILD (구현)

**목적:** 이번 Pulse의 최소 작동 단위를 구현한다.

**실행 순서: 반드시 아래 3단계를 순서대로 완료한다. 에이전트 스폰은 단계 3에서만 허용된다.**

---

#### BUILD 단계 1: PLAN — 경계면 설계

에이전트를 스폰하기 전에 오케스트레이터가 직접 수행한다.

1. 이번 Pulse에서 만들 파일 목록을 작성한다.
2. 파일 간 import/의존 관계를 파악한다.
3. **2개 이상의 파일에서 공유되는 타입/클래스/인터페이스**를 목록화한다.

공유 타입으로 분류하는 기준:
- 파일 A가 파일 B의 클래스·타입을 import하거나 파라미터로 받을 때
- 위젯·컴포넌트가 외부에서 데이터 객체를 props/생성자로 전달받을 때
- 두 모듈이 같은 데이터 구조를 독립적으로 사용할 때

이 목록이 비어 있으면 BUILD 단계 2를 건너뛰고 단계 3으로 진행한다.

---

#### BUILD 단계 2: CONTRACT — 계약서 선(先) 작성

**공유 타입이 1개라도 있으면 반드시 계약서를 먼저 작성한다.**
**계약서 없이 에이전트를 스폰하는 것은 금지한다.**

계약서 작성 방법:
- `pulse-contracts` 모듈이 있으면 해당 스킬을 호출한다.
- 모듈이 없으면 오케스트레이터가 직접 아래 최소 형식으로 작성한다.

최소 계약서 형식 (`_workspace/contracts/{모듈명}.md`):
```
# [모듈명] 인터페이스 계약

## 공유 타입
[클래스·타입·인터페이스 정의 전문]

## 소유 관계
- 정의 파일: [경로]
- 사용 파일: [경로 목록]

## 불변 규칙
- [필드 추가/변경 시 계약서를 먼저 업데이트해야 함]
- [에이전트가 계약서와 다른 타입을 발명하는 것 금지]
```

계약서 작성 완료 후 `_workspace/contracts/README.md`에 목록을 갱신한다.

---

#### BUILD 단계 3: EXECUTE — 에이전트 스폰 및 구현

계약서가 확정된 후에만 구현 에이전트를 스폰한다.

각 에이전트 프롬프트에 반드시 포함:
- `_workspace/contracts/` 경로를 읽고 시작할 것
- "계약서에 정의된 타입을 그대로 사용할 것. 새로운 공유 타입을 발명하지 말 것"
- "계약서에 없는 공유 타입이 필요하면 구현을 중단하고 오케스트레이터에게 보고할 것"

구현 결과를 `_workspace/pulse-N/` 디렉토리에 저장한다.

---

**산출물 구조:**
```
_workspace/pulse-N/
  contracts/          # 이번 Pulse에서 생성한 계약서
  src/                # 구현 코드
  artifacts.md        # 이번 Pulse 산출물 목록
```

---

### 4. VERIFY (검증)

**목적:** 구현이 작동하고 이전 결과를 깨뜨리지 않았음을 확인한다.

**실행 규칙:**
- `pulse-qa` 모듈이 있으면 3-Tier QA를 실행한다:
  - Tier 1: 단위 동작 검증 (빌드, 타입 체크, 단위 테스트)
  - Tier 2: 통합 시나리오 검증 (유저 저니 기반)
  - Tier 3: 비기능 검증 (성능, 보안, 접근성)
- 모듈이 없으면 최소 검증을 실행한다:
  - 빌드 통과 여부
  - 이전 Pulse 결과 비파괴 확인 (회귀 없음)

**검증 실패 처리:**
- 검증 실패 시 즉시 수정한다. 다음 Pulse로 넘기지 않는다.
- 수정 후 같은 VERIFY 단계를 재실행한다.
- 3회 수정 후에도 실패하면 되감기 패턴을 적용한다.

**수렴 감지:**
- 매 VERIFY 단계 후 수렴 여부를 점검한다.
- 상세 기준은 `references/convergence.md`를 참조한다.

---

### 5. LEARN (학습)

**목적:** 이번 Pulse에서 얻은 지식을 기록하고 다음 Pulse에 반영한다.

**실행 규칙:**
- `pulse-immunity` 모듈이 있으면 실패 패턴을 항체로 기록한다.
  - 무엇이 실패했는가, 왜 실패했는가, 어떻게 해결했는가
  - 동일 실패 재발 방지 룰을 자동 생성한다.
- `pulse-dna` 모듈이 있으면 사용자 프로파일을 업데이트한다.
  - 선호 패턴, 결정 스타일, 피드백 경향
- 모듈이 없으면 `_workspace/pulse-log.md`에 다음을 기록한다:

```markdown
## Pulse N — [날짜]

### 발견한 것
- [이번 Pulse에서 새로 알게 된 사실]

### 결정한 것
- [내린 결정과 그 근거]

### 남은 것
- [다음 Pulse에서 해결할 항목]

### 확신도
- 현재 방향에 대한 확신: [0-100%]
```

## AI-Native 패턴 사전

| 패턴 | 설명 | 적용 시점 |
|------|------|----------|
| **분신술** | 같은 에이전트를 다른 관점으로 동시 스폰 | EXPLORE에서 방법이 2개 이상일 때 |
| **시간여행** | 빠른 반복으로 여러 사이클을 돌리기 | 불확실성이 높은 초기 단계 |
| **대립 토론** | 찬성/반대 에이전트로 의사결정 강화 | 중요 아키텍처 결정 시 |
| **되감기** | 잘못된 방향 감지 시 이전 Pulse로 복귀 | VERIFY 3회 실패 또는 방향 오류 감지 시 |
| **탐색 폭발** | 2-3개 방향을 동시 프로토타이핑 | EXPLORE에서 방향이 불명확할 때 |
| **미래 시뮬레이션** | 확장성/유지보수성 사전 검증 | 설계 결정 전 |
| **점진적 확신** | 확신도 낮으면 자동으로 추가 탐색 | EXPLORE 결과 확신도 < 임계값 시 |

## 모듈 연동

Pulse는 독립적으로 동작하지만, 다음 모듈이 설치되어 있으면 각 단계가 강화된다.

| 모듈 | 연동 단계 | 효과 |
|------|----------|------|
| `pulse-interview` | ASK | 구조화된 5-Layer 인터뷰로 요구사항 정제 |
| `pulse-contracts` | BUILD | 인터페이스 계약서 기반으로 에이전트 간 경계면 설정 |
| `pulse-qa` | VERIFY | 3-Tier QA로 검증 깊이 강화 |
| `pulse-immunity` | LEARN | 실패 패턴을 항체로 기록하여 재발 방지 |
| `pulse-dna` | LEARN | 사용자 프로파일 학습으로 개인화 강화 |

각 모듈은 설치 여부에 따라 선택적으로 로드된다. 없어도 기본 동작은 유지된다.

## 토큰 절약 규칙 (필수)

Pulse의 가장 큰 리스크는 **컨텍스트 폭발**이다. 다음 규칙을 반드시 따른다.

### 규칙 1: 파일 기반 통신

에이전트 결과를 메인 컨텍스트에 누적하지 않는다. 반드시 파일로 교환한다.

```
❌ 금지: Agent 결과를 대화에 그대로 포함
✅ 필수: Agent에게 "결과를 _workspace/pulse-N/result.md에 저장하라" 지시
         메인은 완료 알림만 받고, 필요할 때 Read로 특정 파일만 로드
```

모든 Agent 호출 시 프롬프트에 반드시 포함:
- "결과를 `_workspace/pulse-{N}/{agent-name}-result.md`에 Write로 저장하라"
- "대화로 결과를 반환하지 말고 파일 경로만 알려라"

### 규칙 2: 세션 분리 (3~5 Pulse 단위)

하나의 세션에서 모든 Pulse를 처리하지 않는다.

- 3~5 Pulse마다 상태를 `.pulse/state.json`에 저장하고 세션을 끊는다
- 다음 세션에서 state.json을 로드하여 이어간다
- state.json에 저장하는 것: Pulse 번호, 유저 저니 진행률, 미완료 작업, 파일 경로 목록
- state.json에 저장하지 않는 것: 코드 내용, 에이전트 대화 이력, 도구 호출 결과

세션 분리 시점 판단:
- 현재 Pulse가 3 이상이고 LEARN 단계 완료 후 → "세션을 나눌까요?" 제안
- 사용자가 동의하면 상태 저장 후 "다음 세션에서 `pulse로 이어서` 라고 말해주세요" 안내

### 규칙 3: 병렬 에이전트 수 제한

한 번에 스폰하는 에이전트는 **최대 3개**로 제한한다.

```
❌ 금지: 7개 에이전트 동시 스폰
✅ 필수: 최대 3개씩 배치로 나누어 실행
         배치 1 (3개) → 완료 → 배치 2 (3개) → 완료 → ...
```

3개 이상이 필요하면 순차 배치로 나눈다. 각 배치의 결과는 파일로 저장하므로 컨텍스트가 누적되지 않는다.

### 규칙 4: 컨텍스트 경고

다음 징후가 보이면 즉시 세션 분리를 제안한다:
- 에이전트 결과가 길어서 대화에 직접 포함될 때
- 같은 파일을 3회 이상 반복 읽을 때
- 도구 호출이 한 Pulse에서 20회를 초과할 때

## OMC 공존 규칙

oh-my-claudecode(OMC)가 동시에 설치되어 있을 때, Pulse 사이클 중에는 다음 규칙을 따른다:

1. **OMC 모드를 호출하지 않는다** — autopilot, ralph, ultrawork, team 등 OMC 실행 모드는 사용하지 않는다. Pulse 자체 루프(ASK→EXPLORE→BUILD→VERIFY→LEARN)만 사용한다.
2. **OMC 에이전트 카탈로그를 사용하지 않는다** — `oh-my-claudecode:executor` 등의 OMC 에이전트 대신, Claude Code 기본 Agent 도구를 직접 사용한다.
3. **TeamCreate는 Pulse 컨텍스트에서 직접 사용한다** — OMC의 team 파이프라인(team-plan→team-prd→team-exec)이 아닌, Pulse의 BUILD 단계에서 필요 시 직접 TeamCreate를 호출한다.
4. **OMC의 persistent-mode 훅이 Pulse를 차단하면 무시한다** — Pulse의 ASK 단계에서 사용자 응답을 기다리는 것은 정상이며, OMC가 "스킬이 아직 실행 중"이라고 강제하는 것에 따르지 않는다.
5. **OMC 스킬을 트리거하지 않는다** — Pulse 사이클 중 키워드가 OMC 스킬과 겹치더라도(예: "plan", "qa"), OMC 스킬이 아닌 Pulse 모듈을 사용한다.
