# 수렴 감지 기준

Riff 루프가 목표에 도달했는지 판단하는 기준을 정의한다.
매 VERIFY 단계 완료 후 자동으로 체크하며, 수렴이 감지되면 사용자에게 보고한다.

---

## 수렴 판단 지표

### 1. 유저 저니 커버리지

Riff 0에서 정의한 유저 저니 3개 각각에 대해 구현 완료 여부를 추적한다.

| 유저 저니 | 구현 상태 | 검증 통과 |
|----------|----------|----------|
| 저니 1    | [ ] 미완 / [x] 완료 | [ ] / [x] |
| 저니 2    | [ ] 미완 / [x] 완료 | [ ] / [x] |
| 저니 3    | [ ] 미완 / [x] 완료 | [ ] / [x] |

**수렴 기준:** 핵심 저니 100%, 부가 저니 80% 이상 완료

---

### 2. QA 통과율

`riff-qa` 모듈 사용 시 Tier별 통과율을 기준으로 판단한다.

| Tier | 내용 | 수렴 기준 |
|------|------|----------|
| Tier 1 | 빌드, 타입 체크, 단위 테스트 | 100% 통과 |
| Tier 2 | 유저 저니 기반 통합 시나리오 | 95% 이상 통과 |
| Tier 3 | 성능, 보안, 접근성 | 90% 이상 통과 |

`riff-qa` 미설치 시: 빌드 통과 + 회귀 없음으로 대체 적용한다.

---

### 3. 최근 Riff 버그 발생 수

최근 3개 Riff를 기준으로 새로 발견된 버그 수를 추적한다.

| 기간 | 새 버그 수 | 수렴 판단 |
|------|-----------|---------|
| 최근 1개 Riff | 0개 | 우수 |
| 최근 2개 Riff | 0-1개 | 양호 |
| 최근 3개 Riff | 0-2개 | 수렴 임박 |
| 최근 3개 Riff | 3개 이상 | 수렴 아님 |

**수렴 기준:** 최근 3개 Riff 합산 신규 버그 2개 이하

---

### 4. 성공 기준 달성 여부

Riff 0의 `success-criteria.md`에서 정의한 성공 기준 각각의 달성 여부를 확인한다.

```markdown
## 성공 기준 체크리스트

- [ ] 기준 1: [측정 가능한 목표]
- [ ] 기준 2: [측정 가능한 목표]
- [ ] 기준 3: [측정 가능한 목표]
```

**수렴 기준:** 핵심 성공 기준 100%, 부가 기준 80% 이상 달성

---

## Riff별 진행률 추적 테이블

매 Riff 완료 후 아래 테이블을 업데이트한다.

| Riff | 저니 커버리지 | QA 통과율 | 신규 버그 | 성공 기준 달성 | 확신도 | 수렴 여부 |
|-------|------------|---------|---------|-------------|--------|---------|
| 0 | - | - | - | 0% | - | 부팅 |
| 1 | 33% | - | - | 10% | 60% | 진행 중 |
| 2 | 66% | 80% | 2개 | 40% | 72% | 진행 중 |
| 3 | 100% | 95% | 1개 | 80% | 85% | 수렴 임박 |
| 4 | 100% | 98% | 0개 | 100% | 92% | **수렴** |

---

## 수렴 판단 로직

모든 지표를 종합하여 수렴 여부를 결정한다.

```
수렴 = (
  유저 저니 커버리지 >= 95%
  AND QA Tier1 통과율 = 100%
  AND QA Tier2 통과율 >= 95%
  AND 최근 3 Riff 신규 버그 <= 2개
  AND 핵심 성공 기준 달성 = 100%
)
```

---

## 수렴 보고 형식

수렴이 감지되면 다음 형식으로 사용자에게 보고한다.

```markdown
## Riff 수렴 감지

현재까지 [N]개 Riff를 완료했습니다.

**달성 현황:**
- 유저 저니 커버리지: [X]%
- QA 통과율: Tier1 [X]% / Tier2 [X]%
- 최근 3 Riff 신규 버그: [X]개
- 성공 기준 달성: [X]%

**AI 판단:** MVP 완성으로 보입니다.

다음 중 선택해주세요:
1. 완료로 마무리한다
2. 추가 개선 사이클을 돌린다 (어떤 부분을?)
3. 새로운 방향으로 전환한다
```

**반자동 승인 방식:** AI가 수렴을 제안하고, 사용자가 최종 판단한다.
사용자 승인 없이 자동으로 종료하지 않는다.

## 수렴하지 않는 경우 대응

수렴 기준에 도달하지 못한 경우, 상황에 따라 다음을 적용한다.

### 되감기 (Rollback)

**적용 조건:**
- 최근 Riff에서 오히려 품질이 저하된 경우
- 잘못된 방향으로 여러 Riff를 진행한 경우
- VERIFY에서 3회 연속 실패한 경우

**실행 방법:**
1. 마지막으로 안정적이었던 Riff를 식별한다.
2. 해당 Riff의 `_workspace/riff-N/` 상태로 복귀한다.
3. EXPLORE 단계부터 다시 시작하되, 실패 원인을 명시한다.
4. `riff-log.md`에 되감기 이유와 복귀 지점을 기록한다.

```markdown
## 되감기 기록

- 되감기 시점: Riff [N]
- 복귀 지점: Riff [M]
- 이유: [구체적 실패 원인]
- 새 방향: [다음에 시도할 접근]
```

### 방향 전환 제안

**적용 조건:**
- 5개 이상의 Riff를 진행했지만 수렴이 보이지 않는 경우
- 근본적인 가정이 틀린 것으로 판단되는 경우
- 사용자 피드백이 초기 방향과 크게 다른 경우

**실행 방법:**
1. 현재 방향의 문제점을 분석한다.
2. 2-3개의 대안 방향을 제시한다.
3. 각 방향의 예상 소요 Riff와 리스크를 추정한다.
4. 사용자에게 선택을 요청한다.

```markdown
## 방향 전환 제안

현재 방향에서 [N]개 Riff를 진행했으나 수렴 기미가 보이지 않습니다.

**현재 방향의 문제점:**
- [문제 1]
- [문제 2]

**대안 제안:**

| 방향 | 접근 | 예상 Riff | 주요 리스크 |
|------|------|-----------|-----------|
| A    | ...  | 3-4개     | ...       |
| B    | ...  | 5-6개     | ...       |

어떤 방향으로 전환할까요?
```

---

## 수렴 이후

수렴 후 사용자가 완료를 승인하면 다음을 실행한다.

1. `riff-log.md`에 최종 완료 항목을 기록한다.
2. `_workspace/final/` 디렉토리에 최종 산출물을 정리한다.
3. `riff-immunity` 모듈이 있으면 이번 프로젝트의 학습을 항체로 저장한다.
4. `riff-dna` 모듈이 있으면 사용자 프로파일을 최종 업데이트한다.
5. 프로젝트 회고를 간략히 생성한다:
   - 총 Riff 수
   - 주요 결정 지점
   - 가장 유효했던 AI-Native 패턴
   - 다음 프로젝트에 적용할 교훈
