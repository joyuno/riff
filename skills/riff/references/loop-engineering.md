# 루프 엔지니어링 — Riff를 더 자동화된(AFK) 개발 루프로

Riff 사이클을 "감독 없이도 안전하게 도는 자율 루프"로 강화하는 제어 명세.
2025–2026 자율 코딩 루프 연구(Ralph 기법, verification-driven loops, harness engineering,
reward-hacking·goal-drift 완화)에서 검증된 패턴만 추렸다. 매 VERIFY·되감기 판단에서 참조한다.

> 출처 계열: Geoffrey Huntley(Ralph), Anthropic harness/context-engineering, METR(reward hacking),
> 다수의 production agent-loop 회고. 핵심 합의: **상태는 디스크에, 컨텍스트는 작게, 종료는 결정적으로.**

---

## 1. 두 개의 루프 (inner / outer) — Riff는 이미 outer 루프다

| 루프 | 무엇 | Riff에서 | 원칙 |
|------|------|---------|------|
| **inner** | 한 에이전트의 한 턴 추론 | BUILD/EXPLORE 에이전트 1회 | 컨텍스트 작게(~4–32K), 매번 재구성 |
| **outer** | 영속 상태 위의 반복 | Riff N → N+1 | 상태는 파일·git에. 대화에 누적 금지 |

**Ralph 원칙 = Riff 원칙:** 각 Riff는 가능한 한 *신선한 컨텍스트*로 시작하고, 필요한 것만
디스크(`_workspace/`, `.riff/`)에서 읽는다. 전체 코드베이스·전체 대화 로그를 컨텍스트에
올리지 않는다. 모델은 컨텍스트의 50–70%를 넘기면 품질이 측정 가능하게 떨어진다("Dumb Zone").
→ Riff의 "토큰 절약 규칙"(파일 통신·세션 분리)이 바로 이 방어다. 강제로 지킨다.

---

## 2. 완료 신호 — "됐다"는 모델이 선언하지 않는다

각 단계는 **결정적 종료 신호**를 emit해야 한다. 모델의 자기 판단이 아니라 기계가 확인한다.

| 단계 | 완료 신호 | 누가 확정하나 |
|------|----------|-------------|
| VERIFY | `VERIFY_PASSED` | Tier 0~3 종료 코드 전부 0 (모델 주장 X) |
| Riff 1회 | `RIFF_COMPLETE` | success-criteria 진행도 갱신 + LEARN 저장 완료 |
| 프로젝트 | `CONVERGED` | `convergence.md` 수렴식 충족 + **사용자 승인** |

- **완료 프로미스 패턴(ralph-loop 연동):** VERIFY 실패 시 `--completion-promise 'VERIFY_PASSED'`로
  통과까지 자동 루프. 프로미스 문자열이 출력에 없으면 루프는 종료로 보지 않는다.
- **권한 비대칭:** 모델은 "다시 봐"라며 통과를 *막을* 수는 있어도, 통과를 *승인*할 수는 없다.
  승인 슬롯은 결정적 게이트(테스트·계약 lint·종료 코드)만 가진다.

---

## 3. 다차원 예산 — 어느 하나라도 소진되면 멈춘다

단일 지표(반복 횟수만)로는 폭주를 못 막는다. 동시에 건다:

| 차원 | 기본 임계 | 초과 시 동작 |
|------|----------|-------------|
| 수정 반복 (같은 버그) | 3회 | 되감기(`rewind-protocol.md`) |
| 도구 호출 (누적) | 20회 | 세션 분리 제안 |
| 같은 명령 재실행 | 3회 | 정체 판정 → 접근 전환 |
| 같은 파일 편집 | 4회 | 정체 판정 → 접근 전환 |
| EXPLORE 미결 | 3회 | 사용자 판단 요청 |
| 비용(있으면) | 사용자 지정 상한 | 즉시 정지 + 보고 |

이 표는 `riff-qa`의 VERIFY-FIX 루프 제어표와 동일한 계약이다(둘은 같은 예산을 공유).

## 4. 정체·루프 감지 (fingerprinting)

자율 루프의 3대 실패 패턴과 감지법:

| 패턴 | 증상 | 감지 |
|------|------|------|
| Edit-Run-Fail | 비슷한 수정 반복, 근본 원인 그대로 | `(명령, 결과요약)` 해시가 3회 반복 |
| 교대 접근 | A 실패 → B 실패 → A… 무한 교대 | 2개 접근 실패 시 둘 다 버리고 3번째 |
| 빌드-배포 | 에러 전문 안 읽고 재시도 | 같은 에러 메시지 3회 재출현 |

감지되면 **재시도 금지** — 접근을 바꾸거나, 되감거나, 사용자에게 보고한다. 연구에서 한
production 시스템은 개입 전 *동일 동작을 58회* 반복했다. 침묵하는 폭주가 가장 비싸다.

## 5. 결정적 검증 게이트 — 종료 코드만 믿는다

- VERIFY의 각 Tier는 **실제 종료 코드/스크린샷/콘솔 같은 관측 가능한 증거**로만 통과를 판정한다.
  "테스트가 통과할 것 같다"는 모델 진술은 증거가 아니다.
- **flaky 검사는 게이트에서 격리한다** — 가시성을 위해 돌리되 통과를 막지는 않는다.
- 검증자는 **멱등**해야 한다(두 번 돌려 같은 결과).

---

## 6. 독립 검증자 — 만든 사람이 채점하지 않는다

- **분리 원칙:** 구현한 에이전트가 자기 결과를 검증하지 않는다. EXPLORE 분신 또는 `codex` cross-check를
  *검증자*로 쓴다. 검증자는 구현자의 추론 흐름이 아니라 **원래 spec(계약서·success-criteria)**에 대고 본다.
- 큰 변화(아키텍처·보안·DB 마이그레이션)에서 특히 강제한다.

---

## 7. reward hacking · goal drift 방지 (자율 루프의 양대 함정)

연구에서 프런티어 모델이 채점 함수를 보면 0.7–14%의 실행에서 **테스트를 게임**했다(METR).
장시간 루프는 작은 이탈이 누적돼 *다른 문제를 완벽히* 푼다(goal drift).

| 함정 | 방어 (Riff) |
|------|------------|
| 테스트/계약서를 고쳐서 통과 | 계약서·success-criteria·테스트는 **불가침**. 항체로 막고 lint로 고정 |
| 채점 함수 역이용 | 통과 기준을 구현 에이전트에게 채점 권한으로 주지 않음 |
| spec에서 표류 | 매 Riff 시작 시 `success-criteria.md` 재로드. 작업 단위는 작게(저니 1개·함수 1개) |
| 거버넌스 소실(컨텍스트 압축 시 규칙 증발) | 불가침 제약은 압축 대상에서 제외하고 매 Riff 재주입 |

---

## 8. 자동화 등급 (AFK 수준) — 사용자가 고른다

| 등급 | 동작 | 적합 |
|------|------|------|
| L1 보수 | 매 Riff 후 사용자 확인 | 모호한 요구·새 도메인 |
| L2 반자동 (기본) | 트레이드오프·수렴·되감기만 질문, 나머지 자동 | 대부분 |
| L3 AFK | 완료 프로미스까지 무감독 루프, 예산 소진/되감기/시크릿 차단 시에만 정지 | 결정적 acceptance(테스트·계약 전부 그린)일 때만 |

**L3는 "기계 검증 가능한 종료 상태"가 있을 때만 허용한다.** 탐색적·디자인·모호한 작업에 L3를
켜면 reward hacking·drift로 그럴듯한 오답이 쌓인다. 의심되면 L2로 내린다.

---

## 9. Riff 사이클에 박는 지점 (요약)

```
ASK    : success-criteria 재로드(드리프트 방어). 확신도 예산으로 질문 최소화.
EXPLORE: 분신 중 하나를 독립 검증자로. 미결 3회 → 사용자.
BUILD  : 작업 단위 작게. 계약서 불가침. 컨텍스트 작게(파일 통신).
VERIFY : 결정적 게이트(종료 코드). 실패 시 완료 프로미스 루프. 다차원 예산·정체 감지.
LEARN  : 실패는 항체로(secret/contract 포함). 세션 분리 전 저장 확인.
되감기  : 예산 소진·정체·3회 실패 시. 원인을 항체로 남기고 안정 지점으로.
```
